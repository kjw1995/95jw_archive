## 1. 정규화의 개념

정규화(Normalization)는 이상 현상(Anomaly)을 제거하면서 데이터베이스를 올바르게 설계해 나가는 과정이다. 릴레이션을 관련 있는 속성들로만 구성하도록 분해(decomposition)하여 중복을 줄이고 무결성을 확보한다.

```text
비정규형
   │  다중 값 제거
   ▼
  1NF
   │  부분 함수 종속 제거
   ▼
  2NF
   │  이행적 함수 종속 제거
   ▼
  3NF
   │  후보키 아닌 결정자 제거
   ▼
  BCNF
   │  다치 종속 제거
   ▼
  4NF → 5NF
```

실무에서는 보통 3NF 또는 BCNF까지를 목표로 한다.

## 2. 이상 현상

데이터베이스를 잘못 설계하면 불필요한 중복이 생기고, 삽입/수정/삭제 시 다음과 같은 부작용이 발생한다.

### 예제 릴레이션 (잘못된 설계)

```text
┌──────┬──────┬────────┬──────────┬────┐
│ 학번 │ 이름 │과목코드│ 과목명   │성적│
├──────┼──────┼────────┼──────────┼────┤
│ 001  │홍길동│ DB01   │데이터베이스│ A  │
│ 001  │홍길동│ OS01   │운영체제  │ B  │
│ 002  │김영희│ DB01   │데이터베이스│ A  │
│ 003  │이철수│ NET01  │네트워크  │ C  │
└──────┴──────┴────────┴──────────┴────┘
```

### 삽입 이상

새 데이터를 삽입하려고 하는데, 불필요한 데이터까지 억지로 넣어야 하는 문제다.

> "새 과목 'AI01 - 인공지능'을 등록하고 싶다"

수강생이 없으면 학번·이름·성적에 NULL이 들어가야 하는데, 기본키(학번, 과목코드)에 NULL은 허용되지 않으므로 삽입할 수 없다.

### 갱신 이상

중복 투플 중 일부만 수정하여 데이터가 불일치하게 되는 문제다.

> "과목명 '데이터베이스'를 'DB시스템'으로 변경하고 싶다"

한 행만 수정하면 같은 과목코드의 다른 행들과 과목명이 달라져 모순이 발생한다.

### 삭제 이상

투플을 삭제하면 꼭 필요한 정보까지 함께 사라지는 문제다.

> "이철수 학생의 수강 기록을 삭제하고 싶다"

해당 행이 'NET01 - 네트워크'의 유일한 행이면, 학생 삭제와 함께 과목 정보 자체가 사라진다.

## 3. 함수 종속

### 정의

속성 집합 X의 값이 속성 집합 Y의 값을 **유일하게 결정**하면 "X가 Y를 함수적으로 결정한다"고 하며, `X → Y`로 표기한다.

```text
X → Y

X: 결정자 (Determinant)
Y: 종속자 (Dependent)
```

즉, **X 값이 같은 어떤 두 투플도 반드시 Y 값이 같아야 한다**. 이는 릴레이션의 의미(업무 규칙)에서 도출되며, 특정 데이터 스냅샷만 보고 판단해서는 안 된다.

### 판단법

1. 현실 세계의 의미(업무 규칙)에서 "X가 같으면 Y가 반드시 같은가?"를 묻는다.
2. 그렇다면 `X → Y`가 성립한다.
3. 반례 투플이 하나라도 존재할 수 있다면 함수 종속이 아니다.

### 예제

```text
학생 릴레이션
┌──────┬──────┬────────┬──────┐
│ 학번 │ 이름 │학과코드│학과명│
├──────┼──────┼────────┼──────┤
│ 001  │홍길동│ CS     │컴퓨터│
│ 002  │김영희│ BA     │경영  │
│ 003  │이철수│ CS     │컴퓨터│
└──────┴──────┴────────┴──────┘

함수 종속:
  학번 → 이름
  학번 → 학과코드
  학과코드 → 학과명
  학번 → 학과명   (이행적)
```

### 함수 종속 다이어그램

```text
   ┌──────┐
   │ 학번 │
   └──┬───┘
      │
 ┌────┼────┐
 ▼    ▼    ▼
┌──┐ ┌────┐ ┌──┐
│이름│ │학과│ │성적│
└──┘ │코드│ └──┘
     └─┬──┘
       ▼
     ┌────┐
     │학과명│
     └────┘
```

### 완전 함수 종속 vs 부분 함수 종속

| 구분 | 완전 함수 종속 (FFD) | 부분 함수 종속 (PFD) |
|:---|:---|:---|
| 정의 | 결정자 전체가 있어야 종속 성립 | 결정자의 일부만으로도 종속 성립 |
| 조건 | X의 진부분집합으로 결정 불가 | X의 진부분집합으로도 결정 가능 |

```text
수강 릴레이션, 기본키: (학번, 과목코드)

완전 함수 종속:
  (학번, 과목코드) → 성적

부분 함수 종속:
  (학번, 과목코드) → 이름    ← 학번만으로도 결정
  (학번, 과목코드) → 과목명  ← 과목코드만으로도 결정
```

### 이행적 함수 종속

```text
X → Y 이고 Y → Z 이면, X → Z

예시:
  학번 → 학과코드
  학과코드 → 학과명
  ─────────────
  ∴ 학번 → 학과명
```

## 4. 제1정규형 (1NF)

릴레이션의 모든 속성의 도메인이 **원자 값(더 이상 분해할 수 없는 값)** 으로만 구성되어야 한다.

### 1NF 위반

```text
비정규형
┌──────┬──────┬──────────────┐
│ 학번 │ 이름 │수강과목      │
├──────┼──────┼──────────────┤
│ 001  │홍길동│DB, OS, NET   │ ← 다중 값
│ 002  │김영희│DB            │
└──────┴──────┴──────────────┘
```

### 1NF 변환

```text
┌──────┬──────┬────────┐
│ 학번 │ 이름 │수강과목│
├──────┼──────┼────────┤
│ 001  │홍길동│DB      │
│ 001  │홍길동│OS      │
│ 001  │홍길동│NET     │
│ 002  │김영희│DB      │
└──────┴──────┴────────┘

기본키: (학번, 수강과목)
```

## 5. 제2정규형 (2NF)

1NF를 만족하면서, **기본키가 아닌 모든 속성이 기본키에 완전 함수 종속**이어야 한다. 즉 부분 함수 종속이 없어야 한다.

### 2NF 위반

```text
기본키: (학번, 과목코드)

┌──────┬────────┬──────┬──────────┬────┐
│ 학번 │과목코드│ 이름 │ 과목명   │성적│
├──────┼────────┼──────┼──────────┼────┤
│ 001  │ DB01   │홍길동│데이터베이스│ A  │
│ 001  │ OS01   │홍길동│운영체제  │ B  │
│ 002  │ DB01   │김영희│데이터베이스│ A  │
└──────┴────────┴──────┴──────────┴────┘

부분 함수 종속:
  학번 → 이름
  과목코드 → 과목명
```

### 2NF 변환 (분해)

```text
학생               과목
┌──────┬──────┐    ┌────────┬──────────┐
│ 학번 │ 이름 │    │과목코드│ 과목명   │
├──────┼──────┤    ├────────┼──────────┤
│ 001  │홍길동│    │ DB01   │데이터베이스│
│ 002  │김영희│    │ OS01   │운영체제  │
└──────┴──────┘    └────────┴──────────┘

수강
┌──────┬────────┬────┐
│ 학번 │과목코드│성적│
├──────┼────────┼────┤
│ 001  │ DB01   │ A  │
│ 001  │ OS01   │ B  │
│ 002  │ DB01   │ A  │
└──────┴────────┴────┘
```

## 6. 제3정규형 (3NF)

2NF를 만족하면서, **기본키가 아닌 모든 속성이 기본키에 이행적 함수 종속이 되지 않아야** 한다.

### 3NF 위반

```text
┌──────┬──────┬────────┬──────┐
│ 학번 │ 이름 │학과코드│학과명│
├──────┼──────┼────────┼──────┤
│ 001  │홍길동│ CS     │컴퓨터│
│ 002  │김영희│ BA     │경영  │
│ 003  │이철수│ CS     │컴퓨터│
└──────┴──────┴────────┴──────┘

이행적 함수 종속:
  학번 → 학과코드 → 학과명
  ⇒ 학번 → 학과명
```

### 3NF 변환 (분해)

```text
학생                     학과
┌──────┬──────┬──────┐   ┌──────┬──────┐
│ 학번 │ 이름 │학과  │   │학과  │학과명│
│      │      │코드  │   │코드  │      │
├──────┼──────┼──────┤   ├──────┼──────┤
│ 001  │홍길동│ CS   │   │ CS   │컴퓨터│
│ 002  │김영희│ BA   │   │ BA   │경영  │
│ 003  │이철수│ CS   │   └──────┴──────┘
└──────┴──────┴──────┘
```

## 7. 보이스/코드 정규형 (BCNF)

릴레이션에서 **모든 결정자가 후보키**여야 한다. 3NF보다 더 엄격하다.

### 3NF vs BCNF

| 구분 | 조건 |
|:---|:---|
| 3NF | 기본키가 아닌 속성이 기본키에 이행적 종속되지 않음 |
| BCNF | 모든 결정자가 후보키여야 함 |

3NF에서는 "후보키의 일부 속성"이 결정자인 경우를 허용하지만, BCNF는 이마저도 허용하지 않는다.

### BCNF 위반 예제

```text
수강 (한 교수는 한 과목만 담당)
기본키: (학번, 과목), 후보키: (학번, 과목), (학번, 교수)

┌──────┬──────┬──────┐
│ 학번 │ 과목 │ 교수 │
├──────┼──────┼──────┤
│ 001  │ DB   │김교수│
│ 001  │ OS   │이교수│
│ 002  │ DB   │김교수│
│ 003  │ DB   │박교수│
└──────┴──────┴──────┘

함수 종속:
  (학번, 과목) → 교수   (OK, 기본키)
  교수 → 과목            (BCNF 위반)
                        교수는 후보키 아님
```

### BCNF 변환

```text
수강                   교수
┌──────┬──────┐       ┌──────┬──────┐
│ 학번 │ 교수 │       │ 교수 │ 과목 │
├──────┼──────┤       ├──────┼──────┤
│ 001  │김교수│       │김교수│ DB   │
│ 001  │이교수│       │이교수│ OS   │
│ 002  │김교수│       │박교수│ DB   │
│ 003  │박교수│       └──────┴──────┘
└──────┴──────┘

→ 모든 결정자가 후보키
```

## 8. 제4정규형 (4NF)

BCNF를 만족하면서, **다치 종속(MVD, Multi-Valued Dependency)** 이 존재하지 않아야 한다.

### 다치 종속

속성 X에 대해 **여러 Y 값의 집합이 서로 독립적으로 결정**될 때 `X →→ Y`로 표기한다. 하나의 릴레이션에 독립적인 두 다치 종속이 함께 있으면 부적절한 조합이 대거 발생한다.

### 4NF 위반 예제

```text
학생-활동-언어 (학생은 여러 동아리, 여러 언어)

┌──────┬──────┬──────┐
│ 학생 │동아리│ 언어 │
├──────┼──────┼──────┤
│ 001  │축구  │영어  │
│ 001  │축구  │일본어│
│ 001  │음악  │영어  │
│ 001  │음악  │일본어│
└──────┴──────┴──────┘

다치 종속:
  학생 →→ 동아리
  학생 →→ 언어
  (동아리와 언어는 서로 독립)
```

모든 조합이 강제로 생겨 중복이 폭발한다.

### 4NF 변환

```text
학생-동아리            학생-언어
┌──────┬──────┐        ┌──────┬──────┐
│ 학생 │동아리│        │ 학생 │ 언어 │
├──────┼──────┤        ├──────┼──────┤
│ 001  │축구  │        │ 001  │영어  │
│ 001  │음악  │        │ 001  │일본어│
└──────┴──────┘        └──────┴──────┘

→ 독립적인 다치 종속을 각각 분해
```

## 9. 무손실 분해 조건

정규화는 릴레이션을 쪼개는 과정이다. 분해했을 때 원본을 정확하게 복원할 수 있어야 "좋은 분해"다.

### 두 가지 기준

| 기준 | 의미 |
|:---|:---|
| 무손실 분해 (Lossless-Join) | 분해된 릴레이션을 자연 조인하면 원본과 정확히 일치 |
| 종속성 보존 (Dependency Preservation) | 원본의 모든 함수 종속이 분해된 릴레이션들로부터 유도 가능 |

### Heath 정리 (무손실 분해 조건)

릴레이션 R을 속성 집합 R1, R2로 분해했을 때, 다음 조건 중 하나를 만족하면 무손실이다.

```text
R1 ∩ R2 → R1 - R2
         또는
R1 ∩ R2 → R2 - R1
```

즉, **공통 속성이 어느 한쪽 릴레이션의 키 역할**을 하면 무손실이 보장된다.

### 예시

```text
R(학번, 이름, 학과코드, 학과명)
 ├─ R1(학번, 이름, 학과코드)
 └─ R2(학과코드, 학과명)

공통 속성: 학과코드
학과코드 → 학과명 성립
⇒ 학과코드는 R2의 기본키
⇒ 무손실 분해
```

{{< callout type="warning" >}}
BCNF까지 분해하면 무손실은 항상 보장되지만, **종속성 보존은 보장되지 않을 수 있다**. 3NF는 항상 두 조건을 모두 만족하도록 분해할 수 있지만 BCNF는 그렇지 못한 경우가 있다. 그래서 실무에서는 상황에 따라 3NF에서 멈추기도 한다.
{{< /callout >}}

## 10. 역정규화

정규화는 무결성을 보장하지만, 조인이 많아지고 조회 성능이 떨어질 수 있다. 성능을 위해 의도적으로 중복을 허용하는 것을 **역정규화(Denormalization)** 라 한다.

### 대표 기법

| 기법 | 설명 |
|:---|:---|
| 테이블 병합 | 자주 조인되는 테이블을 하나로 합침 |
| 파생 컬럼 추가 | 합계·평균 등 계산 결과를 미리 저장 |
| 중복 컬럼 추가 | 조인 없이 조회하도록 컬럼 복사 |

### 트레이드오프

| 항목 | 정규화 | 역정규화 |
|:---|:---|:---|
| 무결성 | 강함 | 약함 (관리 필요) |
| 저장 공간 | 절약 | 증가 |
| 조회 성능 | 낮음 (조인 많음) | 높음 |
| 갱신 이상 | 없음 | 발생 가능 |

## 핵심 정리

| 개념 | 설명 |
|:---|:---|
| 이상 현상 | 삽입·갱신·삭제 이상 |
| 함수 종속 | X → Y, 업무 규칙에서 도출 |
| 완전 / 부분 FD | 결정자 전체 / 일부에 종속 |
| 이행적 FD | X→Y, Y→Z ⇒ X→Z |
| 1NF | 모든 속성이 원자 값 |
| 2NF | 1NF + 부분 FD 제거 |
| 3NF | 2NF + 이행적 FD 제거 |
| BCNF | 모든 결정자가 후보키 |
| 4NF | BCNF + 다치 종속 제거 |
| 무손실 분해 | 공통 속성이 한쪽의 키 역할 |
| 역정규화 | 성능을 위해 중복을 허용 |

{{< callout type="info" >}}
**용어 정리**
- **정규화**: 이상 현상을 제거하며 릴레이션을 분해하는 과정
- **이상 현상**: 삽입·갱신·삭제 시 발생하는 부작용
- **함수 종속 (FD)**: X 값이 Y 값을 유일하게 결정하는 관계
- **결정자 / 종속자**: FD에서 왼쪽(X) / 오른쪽(Y) 속성
- **완전 / 부분 함수 종속**: 결정자 전체 / 일부에 종속
- **이행적 함수 종속**: X→Y, Y→Z에서 파생되는 X→Z
- **다치 종속 (MVD)**: 하나의 X에 여러 Y 값 집합이 독립적으로 존재
- **1NF / 2NF / 3NF / BCNF / 4NF**: 점점 엄격해지는 정규형
- **무손실 분해**: 자연 조인으로 원본을 완벽 복원 가능한 분해
- **종속성 보존**: 원본 FD들을 분해 후에도 유도 가능
- **역정규화**: 성능 향상을 위해 중복을 의도적으로 허용
{{< /callout >}}
