## 1. 데이터 모델링 개요

데이터 모델링은 현실 세계의 데이터를 데이터베이스로 옮기기 위한 변환 과정이다. 현실 세계에서 식별한 개체와 관계를 개념적 구조로 표현한 뒤 DBMS가 지원하는 논리적 구조로 변환한다.

```text
 현실 세계      개념 세계      컴퓨터 세계
┌────────┐   ┌────────┐    ┌────────┐
│ 고객   │   │ E-R    │    │ TABLE  │
│ 상품   │──→│ 다이어 │───→│ CREATE │
│ 주문   │   │ 그램   │    │ 스키마 │
└────────┘   └────────┘    └────────┘
    │            │              │
    ▼            ▼              ▼
 요구사항    개념적 구조    논리적 구조
  분석      (DBMS 독립)    (DBMS 종속)
```

### 모델링 단계

| 단계 | 설명 | 산출물 | 도구 |
|:---|:---|:---|:---|
| 개념적 모델링 | 현실 → 개념 세계 | 개념적 구조 | E-R 모델 |
| 논리적 모델링 | 개념 → 논리적 구조 | 스키마 | 관계 데이터 모델 |
| 물리적 모델링 | 논리 → 저장 구조 | 물리 스키마 | DBMS 설정 |

### 데이터 모델의 3요소

| 요소 | 특성 | 내용 |
|:---|:---|:---|
| 데이터 구조 (Structure) | 정적 | 개체, 속성 |
| 연산 (Operation) | 동적 | CRUD 작업 |
| 제약조건 (Constraint) | 규칙 | 무결성, 데이터 제한 |

## 2. E-R 모델과 기본 기호

E-R 모델은 피터 첸(1976)이 제안한 표기법으로, 개체와 관계로 현실 세계를 표현한다.

```text
┌──────┐   개체 (Entity)
│ 고객 │   사각형, 식별 가능한 사물
└──────┘

 ◇────◇    관계 (Relationship)
 │주문│    마름모, 개체 간 연관
 ◇────◇

 (이름)    속성 (Attribute)
           타원, 특성

 (_고객ID_)  키 속성, 밑줄

 ((연락처))  다중 값 속성, 이중 타원

 (- 나이 -)  유도 속성, 점선 타원
```

## 3. 개체 (Entity)

개체는 현실 세계에서 구별 가능한 사람, 사물, 개념이다. 개체 타입은 속성들의 집합을 정의한 틀이고, 개체 인스턴스는 실제 값으로 채워진 하나의 데이터다.

```text
개체 타입
┌─────────────────────────────┐
│ 고객(이름,연락처,주소,등급) │
└──────────────┬──────────────┘
               ▼
개체 인스턴스
┌─────────────────────────────┐
│ 홍길동, 010-1234, 서울, VIP │
│ 김철수, 010-5678, 부산, 일반 │
│ 이영희, 010-9999, 대전, VIP │
└─────────────────────────────┘
               │
               ▼
   개체 집합 (Entity Set)
```

### 용어 대응

| E-R 모델 | 파일 시스템 | 관계형 DB |
|:---|:---|:---|
| 개체 타입 | 레코드 타입 | 테이블 (릴레이션) |
| 개체 인스턴스 | 레코드 인스턴스 | 행 (튜플) |
| 속성 | 필드 | 컬럼 (속성) |

## 4. 속성 (Attribute)

속성은 개체가 가지는 고유한 특성이다. 값의 개수, 분해 가능 여부, 계산 여부에 따라 분류한다.

### 속성 유형

| 유형 | 설명 | 예시 | E-R 기호 |
|:---|:---|:---|:---|
| 단일 값 | 하나의 값 | 이름, 주민번호 | ( ) |
| 다중 값 | 여러 값 허용 | 연락처, 취미 | (( )) |
| 단순 | 분해 불가 | 나이, 성별 | ( ) |
| 복합 | 여러 속성으로 분해 | 주소 → 시·구·동 | ( )─( ) |
| 유도 | 다른 속성에서 계산 | 나이 (생년월일) | (- -) |
| 저장 | 실제 저장 값 | 생년월일, 정가 | ( ) |
| 널 | NULL 허용 | 배우자명 | ( ) |
| 키 | 식별자 | 고객ID, 주민번호 | ( _ _ ) |

### 고객 개체의 속성 예시

```text
         ┌──────┐
         │ 고객 │
         └──┬───┘
 ┌─────┬───┼────┬──────┐
 │     │   │    │      │
(_ID_)(이름)((연락처))(주소)(-나이-)
  키  단일값 다중값   │   유도
                     │
              ┌──┬───┼──┬───┐
             (시)(구)(동)(상세)
                 복합 속성
```

## 5. 관계 (Relationship)

관계는 개체와 개체 간의 의미 있는 연관성이다. 참여 개체 수에 따라 이항·삼항·순환으로 나뉜다.

### 참여 개체 수

```text
이항 관계 (Binary)
┌──────┐  ◇────◇  ┌──────┐
│ 고객 │──│주문│──│ 상품 │
└──────┘  ◇────◇  └──────┘

삼항 관계 (Ternary)
      ┌──────┐
      │ 고객 │
      └───┬──┘
          │
  ┌───────◇───────┐
  │     구매      │
  └───┬───────┬───┘
  ┌───┴──┐ ┌──┴───┐
  │ 상품 │ │ 매장 │
  └──────┘ └──────┘

순환 관계 (Recursive)
┌──────┐
│ 사원 │◇──┐ 관리
└───┬──┘   │
    └──────┘
```

### 매핑 카디널리티 (Cardinality)

| 유형 | 예시 | 의미 |
|:---|:---|:---|
| 일대일 (1:1) | 사원 ─ 사물함 | 각 사원이 사물함 1개, 각 사물함에 사원 1명 |
| 일대다 (1:N) | 부서 ─ 사원 | 한 부서에 여러 사원, 사원은 한 부서 소속 |
| 다대다 (M:N) | 학생 ─ 과목 | 학생이 여러 과목, 과목에 여러 학생 |

### Chen 표기법 vs Crow's Foot 표기법

두 표기법은 카디널리티를 다른 기호로 나타낸다.

```text
 Chen              Crow's Foot
────────           ─────────────
1:1  (1,1)         ──┼┼──
1:N  (1,N)         ──┼<──
M:N  (M,N)         ──><──
선택 참여          ──○──
필수 참여          ──┼──
```

| 표기법 | 카디널리티 표현 | 특징 |
|:---|:---|:---|
| Chen | 선 위에 숫자/문자 (1, N, M) | 개념 설계에 적합, 관계 속성 표현 쉬움 |
| Crow's Foot | 선 끝 모양 (까마귀 발, 막대, 원) | 논리·물리 설계에 널리 쓰임 |

### 참여 특성

```text
필수 참여 (전체 참여) ═══
모든 개체 인스턴스가 관계에 참여

┌──────┐         ┌──────┐
│ 사원 │══◇══════│ 부서 │
└──────┘  소속   └──────┘

선택 참여 (부분 참여) ───
일부 인스턴스만 관계에 참여

┌──────┐         ┌──────┐
│ 고객 │──◇──────│ 주문 │
└──────┘  구매   └──────┘
```

### 강한 개체와 약한 개체

```text
┌──────┐         ╔════════╗
│ 직원 │══◇◇═════║ 부양가족 ║
└──────┘  부양   ╚════════╝
강한 개체         약한 개체
                (직원 없이 존재 불가)
```

- 강한 개체: 자신의 키로 독립 식별. 일반 사각형.
- 약한 개체: 강한 개체에 종속. 이중 사각형. 부분 키(구별자) + 강한 개체 키로 식별.

## 6. E-R 다이어그램 실전 예제

### 쇼핑몰 E-R 다이어그램

```text
 (_고객ID_) (이름) (연락처)
       \    |    /
       ┌─────────┐
       │  고객   │
       └────┬────┘
            │ 1
       ◇────┴────◇
       │  주문   │──(주문일자)
       ◇────┬────◇
            │ N
       ┌────┴────┐
       │주문상세 │──(수량)
       └────┬────┘
            │ N
       ◇────┴────◇
       │  포함   │
       ◇────┬────◇
            │ 1
       ┌────┴────┐
       │  상품   │
       └─────────┘
       /    |    \
 (_상품ID_)(상품명)(가격)(재고)
```

관계 정리는 다음과 같다.

- 고객 : 주문 = 1 : N (한 고객이 여러 주문)
- 주문 : 상품 = M : N (주문상세로 해소)

## 7. E-R 다이어그램 → 릴레이션 변환 규칙

E-R 다이어그램은 아래 규칙으로 관계형 테이블로 변환한다.

| E-R 요소 | 관계형 요소 | 변환 규칙 |
|:---|:---|:---|
| 개체 | 테이블 | 개체 타입 → 테이블 |
| 속성 | 컬럼 | 속성 → 컬럼 |
| 키 속성 | 기본 키 | PRIMARY KEY 지정 |
| 다중 값 속성 | 별도 테이블 | 새 테이블로 분리 |
| 복합 속성 | 분해 또는 유지 | 하위 속성으로 분해 |
| 1:1 관계 | 외래 키 | 어느 한쪽에 FK |
| 1:N 관계 | 외래 키 | N쪽에 FK |
| M:N 관계 | 연결 테이블 | 양쪽 FK를 가진 새 테이블 |
| 관계 속성 | 컬럼 | FK 있는 테이블 또는 연결 테이블에 배치 |
| 약한 개체 | 복합 키 | 강한 개체 PK + 구별자 |

### 변환 SQL 예시

```sql
-- 개체 → 테이블
CREATE TABLE 고객 (
    고객ID INT PRIMARY KEY,
    이름 VARCHAR(50) NOT NULL,
    연락처 VARCHAR(20)
);

CREATE TABLE 상품 (
    상품ID INT PRIMARY KEY,
    상품명 VARCHAR(100) NOT NULL,
    가격 DECIMAL(10,2) NOT NULL,
    재고 INT DEFAULT 0
);

-- 1:N 관계 → N쪽에 FK
CREATE TABLE 주문 (
    주문ID INT PRIMARY KEY,
    고객ID INT NOT NULL,
    주문일자 DATE NOT NULL,
    FOREIGN KEY (고객ID) REFERENCES 고객(고객ID)
);

-- M:N 관계 → 연결 테이블
CREATE TABLE 주문상세 (
    주문ID INT,
    상품ID INT,
    수량 INT NOT NULL,
    PRIMARY KEY (주문ID, 상품ID),
    FOREIGN KEY (주문ID) REFERENCES 주문(주문ID),
    FOREIGN KEY (상품ID) REFERENCES 상품(상품ID)
);
```

### 다중 값 속성 변환

```sql
-- 잘못된 방법 (1NF 위반)
CREATE TABLE 고객_잘못 (
    고객ID INT PRIMARY KEY,
    이름 VARCHAR(50),
    연락처 VARCHAR(100)  -- "010-1234, 010-5678" ?
);

-- 올바른 방법: 별도 테이블 분리
CREATE TABLE 고객 (
    고객ID INT PRIMARY KEY,
    이름 VARCHAR(50)
);

CREATE TABLE 고객연락처 (
    고객ID INT,
    연락처 VARCHAR(20),
    연락처유형 VARCHAR(10),
    PRIMARY KEY (고객ID, 연락처),
    FOREIGN KEY (고객ID) REFERENCES 고객(고객ID)
);
```

{{< callout type="warning" >}}
다중 값 속성을 한 컬럼에 콤마로 이어 넣으면 1NF를 위반한다. 검색·인덱싱·무결성 검사가 모두 깨지므로 반드시 별도 테이블로 분리해야 한다.
{{< /callout >}}

## 8. 논리적 데이터 모델

E-R 다이어그램을 DBMS가 지원하는 논리 구조로 변환한 것이 논리적 데이터 모델이다.

| 모델 | 구조 | 사용 현황 |
|:---|:---|:---|
| 관계 데이터 모델 | 2차원 테이블 | 가장 널리 쓰임 (MySQL, Oracle 등) |
| 계층 데이터 모델 | 트리 구조 | 초기 DB 모델 (IMS) |
| 네트워크 데이터 모델 | 그래프 구조 | 현재 거의 안 씀 |

### 관계 데이터 모델 구조

```text
고객 테이블                  주문 테이블
┌─────┬──────┬─────────┐    ┌─────┬─────┬──────┐
│ID   │이름  │연락처   │    │주문 │고객 │일자  │
├─────┼──────┼─────────┤    ├─────┼─────┼──────┤
│ 1   │홍길동│010-1234 │←───│101  │ 1   │24/1/1│
│ 2   │김철수│010-5678 │←───│102  │ 1   │24/1/5│
│ 3   │이영희│010-9999 │    │103  │ 2   │24/1/7│
└─────┴──────┴─────────┘    └─────┴─────┴──────┘
 PK                          PK    FK
```

{{< callout type="info" >}}
실무 체크리스트: 키 속성 지정, 다중 값 속성은 별도 테이블로, 카디널리티 정확 파악(1:1/1:N/M:N), 참여 특성(필수/선택) 확인, M:N은 연결 테이블로 분해, 약한 개체는 복합 키 처리.
{{< /callout >}}

## 핵심 정리

| 개념 | 설명 |
|:---|:---|
| 데이터 모델링 | 현실 → 개념 → 논리 → 물리 |
| E-R 모델 | 개체, 속성, 관계로 현실 표현 |
| 개체 (Entity) | 사각형, 구별 가능한 사물 |
| 속성 (Attribute) | 타원, 개체의 특성 |
| 관계 (Relationship) | 마름모, 개체 간 연관 |
| 카디널리티 | 1:1, 1:N, M:N |
| Chen / Crow's Foot | 개념 설계 / 논리·물리 설계 표기법 |
| 약한 개체 | 강한 개체에 종속, 복합 키로 식별 |
| 1:N 변환 | N쪽에 외래 키 |
| M:N 변환 | 연결 테이블 생성 |

{{< callout type="info" >}}
**용어 정리**
- **데이터 모델링**: 현실 세계의 데이터를 DB 구조로 옮기는 과정
- **E-R 모델**: 개체와 관계로 현실을 표현하는 개념적 모델
- **개체 (Entity)**: 현실에서 구별 가능한 사람·사물·개념
- **속성 (Attribute)**: 개체의 고유한 특성
- **관계 (Relationship)**: 개체 간의 의미 있는 연관성
- **카디널리티 (Cardinality)**: 관계에 참여하는 개체 수 (1:1, 1:N, M:N)
- **Chen 표기법**: 숫자·문자로 카디널리티 표기
- **Crow's Foot 표기법**: 선 끝 모양으로 카디널리티 표기
- **키 속성 (Key Attribute)**: 개체를 식별하는 속성
- **다중 값 속성**: 여러 값을 가질 수 있는 속성
- **유도 속성**: 다른 속성에서 계산되는 속성
- **약한 개체 (Weak Entity)**: 다른 개체에 종속된 개체
- **구별자 (Discriminator)**: 약한 개체의 부분 키
- **스키마 (Schema)**: 데이터베이스의 논리적 구조
{{< /callout >}}
