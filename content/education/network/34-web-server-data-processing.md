## 1. 개요

웹 서버에 도착한 전기 신호는 OSI 하위 계층부터 상위 계층으로 올라가며 역캡슐화된다. 각 계층에서 헤더를 벗겨 내고 해당 계층의 검증을 거친 뒤 상위 계층으로 전달한다. 응답을 보낼 때는 반대로 응용 → 전송 → 네트워크 → 데이터링크 → 물리 순서로 다시 캡슐화한다.

```
수신                      송신
[물리]      비트       ← 비트
   ▲                      │
[데이터링크] 프레임    ← 프레임
   ▲                      │
[네트워크]  패킷       ← 패킷
   ▲                      │
[전송]      세그먼트   ← 세그먼트
   ▲                      │
[응용]      HTTP 요청  → HTTP 응답
```

## 2. 계층별 수신 처리

### 물리 계층

- NIC가 전기 신호를 비트로 변환
- 프리앰블로 동기화
- 신호 품질(잡음, 감쇠) 확인

### 데이터링크 계층

```
┌──────────┬──────┬──────┐
│ Ethernet │ IP+  │ FCS  │
│ 헤더     │ TCP+ │      │
│          │ HTTP │      │
└──────────┴──────┴──────┘
```

- 목적지 MAC이 자신인지 확인 (아니면 폐기)
- FCS(CRC-32) 검증, 손상 시 폐기
- 이더넷 타입 확인 (IPv4=0x0800)
- 이더넷 헤더·FCS 제거 후 상위 계층 전달

### 네트워크 계층

- 목적지 IP가 자신인지 확인
- IP 헤더 체크섬 검증
- TTL > 0 확인
- 프로토콜 필드로 다음 계층 결정 (6=TCP, 17=UDP, 1=ICMP)
- IP 헤더 제거

### 전송 계층 (TCP)

```
1) 목적지 포트로 애플리케이션 식별
   (80 → HTTP, 443 → HTTPS)
2) TCP 체크섬 검증
3) 시퀀스 번호 확인, 재정렬/중복 제거
4) ACK 번호 계산 후 응답 준비
5) TCP 헤더 제거
```

### 응용 계층

HTTP 요청 메시지를 파싱해 메서드, URI, 헤더, 본문을 해석한다.

```
GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Accept: text/html
Accept-Encoding: gzip
```

1. 메서드 판별 (GET, POST 등)
2. URI 파싱 → 파일 경로 또는 라우팅
3. 헤더 파싱 (가상 호스트, 언어, 인코딩 협상)
4. 리소스 조회 또는 동적 처리
5. 응답 상태 코드와 본문 생성

## 3. 응답 캡슐화

서버는 응답 HTTP 메시지를 생성한 뒤, 응용 → 전송 → 네트워크 → 데이터링크 순으로 헤더를 붙여 내보낸다.

```
응용      HTTP/1.1 200 OK + HTML
   │ + TCP 헤더 (src 80, dst 54321)
   ▼
전송      세그먼트
   │ + IP 헤더
   ▼
네트워크  패킷
   │ + 이더넷 헤더 + FCS
   ▼
데이터링크 프레임
   │ NIC
   ▼
물리      전기 신호
```

## 4. 웹 서버 소프트웨어

| 제품 | 특징 |
|:---|:---|
| Apache HTTPD | 모듈 기반, `.htaccess`, MPM(prefork/worker/event) |
| Nginx | 이벤트 기반 비동기, 역방향 프록시/로드밸런서 |
| Microsoft IIS | Windows 표준, ASP.NET, AD 통합 |
| Node.js (Express) | 단일 스레드 이벤트 루프, JS 애플리케이션 |

### 처리 모델

```
Prefork (프로세스 기반)
[요청 1] → [프로세스 1]
[요청 2] → [프로세스 2]

Worker/Event (스레드·이벤트 기반)
[프로세스]
  ├ 스레드 1 ← 요청 1
  ├ 스레드 2 ← 요청 2
  └ 이벤트 루프 ← 다수 연결
```

{{< callout type="info" >}}
정적 콘텐츠 응답과 동적 처리는 역할을 나누는 편이 흔하다. 앞단에는 Nginx가 서서 정적 파일을 직접 응답하고 Keep-Alive·TLS 종단을 처리하며, 동적 요청은 뒤쪽 WAS(Spring, Node, Django 등)로 프록시하는 구성이 일반적이다.
{{< /callout >}}

## 5. 라우팅과의 관계

웹 서버로 향하는 패킷은 수많은 라우터를 거친다. 서버 쪽 라우팅은 보통 아래 두 방식을 섞어 쓴다.

| 구분 | 정적 라우팅 | 동적 라우팅 |
|:---|:---|:---|
| 설정 | 수동 | 프로토콜이 자동 |
| 장애 우회 | 수동 | 자동 |
| 대역폭 사용 | 없음 | 제어 메시지 교환 |
| 적합 규모 | 소규모, 기본 경로 | 대규모 네트워크 |
| 대표 프로토콜 | - | RIP, OSPF, EIGRP, BGP |

### 대표 동적 라우팅 프로토콜

| 프로토콜 | 분류 | 메트릭 | 특징 |
|:---|:---|:---|:---|
| RIP | IGP, 거리 벡터 | 홉 수 (최대 15) | 소규모 |
| OSPF | IGP, 링크 상태 | 대역폭 기반 비용 | 빠른 수렴, Area 구조 |
| EIGRP | IGP, 하이브리드 | 복합 메트릭 | Cisco 독점 |
| BGP | EGP, 경로 벡터 | 경로 속성/정책 | 인터넷 AS 간 라우팅 |

## 6. 실무 도구

### 라우팅 테이블

```bash
ip route show
ip route get 93.184.216.34
route -n
```

Cisco 장비에서는 `show ip route`로 현재 라우팅 테이블과 각 경로의 출처(C: 직접, S: 정적, O: OSPF, B: BGP)를 볼 수 있다.

### 관리 거리(Administrative Distance)

여러 출처의 동일 목적지 경로 중 어느 것을 쓸지 결정하는 값이다. 낮을수록 신뢰도가 높다.

| 출처 | AD |
|:---|:---|
| 직접 연결 | 0 |
| 정적 라우트 | 1 |
| EIGRP (내부) | 90 |
| OSPF | 110 |
| RIP | 120 |
| BGP (외부) | 200 |

### 웹 서버 점검

```bash
# Apache/Nginx 프로세스 확인
ps aux | grep -E 'httpd|nginx'
systemctl status nginx

# 리스닝 포트
ss -tlnp
netstat -tlnp | grep :80

# 실시간 로그
tail -f /var/log/nginx/access.log
tail -f /var/log/nginx/error.log
```

## 핵심 정리

| 항목 | 내용 |
|:---|:---|
| 수신 처리 | 물리 → 데이터링크 → 네트워크 → 전송 → 응용 순 역캡슐화 |
| 응답 처리 | 응용 → 전송 → 네트워크 → 데이터링크 → 물리 순 캡슐화 |
| 각 계층 검증 | MAC 일치, FCS, IP 체크섬, TTL, TCP 체크섬 |
| 서버 구조 | 정적 응답 + 동적 처리 분리, 이벤트/스레드 기반 모델 |
| 라우팅 | 소규모는 정적, 대규모는 OSPF/BGP 같은 동적 프로토콜 |

{{< callout type="info" >}}
**용어 정리**
- **역캡슐화**: 수신 시 각 계층이 자신의 헤더를 제거하고 상위로 전달
- **캡슐화**: 송신 시 각 계층이 헤더를 덧붙여 하위로 전달
- **MPM**: Apache의 Multi-Processing Module, 요청 처리 방식(prefork/worker/event)
- **Administrative Distance**: 라우팅 출처별 신뢰도
- **Longest Prefix Match**: 가장 구체적인 경로가 선택되는 라우팅 규칙
{{< /callout >}}
