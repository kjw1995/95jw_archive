동시성 프로그램을 만들려면 가장 먼저 **문제를 동시에 실행할 수 있는 작은 작업으로 쪼개야** 한다. 이 과정을 **분해(decomposition)** 라 한다. 문제를 분해하는 만능 공식은 없지만, 작업 사이의 의존 관계를 그래프로 그려 보면 "무엇을 동시에 돌릴 수 있는가"가 드러난다. 이 장에서는 분해의 출발점인 **작업 의존 관계 그래프**, 두 갈래 전략인 **작업 분해(Task)** 와 **데이터 분해(Data)**, 널리 쓰이는 패턴인 **파이프라인 / 맵 / 포크·조인 / 맵·리듀스**, 그리고 분해의 알갱이 크기를 정하는 **입도(granularity)** 를 살펴본다.

---

## 1. 왜 분해가 필요한가

### 1.1 동시성의 출발점

동시성 프로그래밍의 첫 단추는 알고리즘이나 동기화가 아니라 **분해**다. 하나의 큰 문제를 여러 처리 자원(코어·스레드·프로세스)에 나눠 주려면, 먼저 그 문제를 **동시에 실행 가능한 작업 단위**로 나눠야 하기 때문이다.

> **분해(decomposition)** 란 "문제를 동시에 실행할 수 있는 작업으로 나누려면 어떻게 쪼개야 하는가?"라는 물음에 답을 찾는 과정이다.

```
┌──────────────────────────────────────────────────────────────┐
│         동시성 프로그램을 만드는 흐름                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   ① 분해         문제 → 동시 실행 가능한 작업들              │
│        │                                                     │
│        ▼                                                     │
│   ② 의존 분석    작업 간 의존 관계 파악 (DAG)               │
│        │                                                     │
│        ▼                                                     │
│   ③ 매핑         작업을 스레드·프로세스·코어에 배치          │
│        │                                                     │
│        ▼                                                     │
│   ④ 동기화       공유 자원 보호 (다음 장)                    │
│                                                              │
│   이 장은 ①~②, 즉 "어떻게 나눌까"를 다룬다                  │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 만능 공식은 없다

문제를 분해하는 **단일한 정답 공식은 존재하지 않는다.** 같은 문제도 자원 구성, 데이터 형태, 의존 관계에 따라 최적 분해가 달라진다. 다만 출발점은 늘 같다 — **작업 사이의 의존 관계를 먼저 그려 보는 것**이다. 의존 관계를 알아야 어떤 작업을 진짜로 동시에 돌릴 수 있는지 판단할 수 있다.

{{< callout type="info" >}}
분해는 **무엇을(what)** 나눌지를 정하는 단계이고, 매핑은 **어디서(where)** 실행할지를 정하는 단계다. 둘을 섞어 생각하면 "코어가 4개니까 4조각"처럼 자원에 끌려가는 분해를 하게 된다. 먼저 문제의 구조(의존 관계)를 보고 나눈 뒤, 자원에 맞춰 입도를 조정하는 순서가 좋다.
{{< /callout >}}

## 2. 작업 의존 관계 그래프

### 2.1 정의

> **작업 의존 관계 그래프(task dependency graph)** 는 작업을 노드로, 작업 간 "먼저 끝나야 한다"는 의존 관계를 방향 간선으로 표현한 **방향성 비순환 그래프(DAG)** 다.

화살표가 `A → B`이면 "A가 끝나야 B를 시작할 수 있다"는 뜻이다. 화살표로 이어지지 **않은** 작업들은 서로 의존하지 않으므로 **동시에 실행할 수 있다.** 즉 이 그래프는 동시 실행 가능 지점을 한눈에 보여 준다.

```
┌──────────────────────────────────────────────────────────────┐
│         작업 의존 관계 그래프 (DAG)                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│            ┌────┐                                            │
│            │ A  │  시작 작업                                  │
│            └─┬──┘                                            │
│         ┌────┴────┐                                          │
│         ▼         ▼                                          │
│      ┌────┐    ┌────┐   B·C는 서로 의존 없음                 │
│      │ B  │    │ C  │   → 동시 실행 가능                     │
│      └─┬──┘    └─┬──┘                                        │
│         └───┬────┘                                           │
│             ▼                                                │
│          ┌────┐    D는 B·C가 모두 끝나야 시작                 │
│          │ D  │                                              │
│          └────┘                                              │
│                                                              │
│  임계 경로(critical path) = 가장 긴 의존 사슬               │
│   → 아무리 코어가 많아도 이 길이보다 빨라질 수 없다          │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 임계 경로와 병렬화 한계

그래프에서 **가장 긴 의존 사슬**을 **임계 경로(critical path)** 라 한다. 이 경로 위의 작업들은 반드시 순서대로 실행되어야 하므로, **임계 경로의 길이가 전체 실행 시간의 하한**이 된다. 코어를 아무리 늘려도 임계 경로보다 빠를 수는 없다 — 이것이 2장에서 본 **암달의 법칙**이 분해 단계에서 나타나는 모습이다.

| 그래프에서 읽는 것 | 의미 |
|:------------------|:-----|
| 화살표 없는 노드들 | 동시 실행 가능한 작업 |
| 임계 경로 길이 | 병렬화로도 줄일 수 없는 최소 시간 |
| 분기 폭(너비) | 동시에 활용 가능한 최대 병렬도 |
| 합류 지점 | 동기화·대기(join)가 필요한 지점 |

{{< callout type="info" >}}
의존 관계 그래프는 머릿속으로만 그려도 충분할 때가 많다. 빌드 시스템(Make, Gradle), 데이터 파이프라인(Airflow), 비동기 조합(`CompletableFuture.thenCombine`)이 모두 내부적으로 이 DAG를 만들어 "독립한 작업은 병렬로, 의존하는 작업은 순서대로" 실행한다.
{{< /callout >}}

## 3. 작업 분해 (Task Decomposition)

### 3.1 서로 다른 '일'을 나눈다

> **작업 분해**란 "문제를 동시 실행 가능한 작업으로 나누려면 어떻게 분해해야 하는가?"의 답을 찾는 과정으로, 애플리케이션을 **기능적으로 독립적인 작업**들로 쪼개는 전략이다.

애플리케이션에 명확히 구분되는 **기능별 구성 요소**가 있다면, 각 기능을 독립 작업으로 떼어 내 동시에 실행한다. 예를 들어 대시보드를 만들 때 "매출 집계", "사용자 통계", "로그 분석"은 서로 다른 일이고 의존도 없으므로 동시에 돌릴 수 있다.

이는 서로 **다른 명령 흐름**이 **다른 데이터**를 다루는 형태이므로, 3장 플린 분류의 **MIMD**(여러 명령·여러 데이터)에 해당한다. 같은 데이터를 여러 단계가 거쳐 가는 파이프라인이라면 **MISD** 성격을 띤다.

```
┌──────────────────────────────────────────────────────────────┐
│         작업 분해 — 기능별로 쪼갠다                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│              [대시보드 만들기]                                │
│           ┌────────┼────────┐                                │
│           ▼        ▼        ▼                                │
│        ┌──────┐ ┌──────┐ ┌──────┐                           │
│        │ 매출  │ │ 사용자│ │ 로그  │   서로 다른 '일'         │
│        │ 집계  │ │ 통계  │ │ 분석  │   → 동시 실행            │
│        └──────┘ └──────┘ └──────┘                           │
│           기능적으로 독립 (MIMD)                              │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 Java로 작업 분해

`CompletableFuture`는 독립적인 기능 작업을 별도 스레드에 맡기고 마지막에 합류시키기에 알맞다.

```java
// 작업 분해 — 서로 다른 기능을 독립 작업으로 동시에
CompletableFuture<Report> sales =
    CompletableFuture.supplyAsync(() -> buildSalesReport());   // 매출
CompletableFuture<Report> users =
    CompletableFuture.supplyAsync(() -> buildUserStats());     // 사용자
CompletableFuture<Report> logs  =
    CompletableFuture.supplyAsync(() -> analyzeLogs());        // 로그

// 세 작업이 모두 끝날 때까지 합류(join) — DAG의 합류 지점
CompletableFuture.allOf(sales, users, logs).join();
Dashboard dash = new Dashboard(sales.join(), users.join(), logs.join());
```

{{< callout type="info" >}}
작업 분해의 병렬도는 **기능의 개수**에 묶인다. 기능이 3개면 3개를 넘어 병렬화할 수 없다 — 더 잘게 쪼개려면 각 기능 *안의 데이터*를 나누는 데이터 분해(5장 본문)와 결합해야 한다. 실무에서는 두 전략을 함께 쓰는 경우가 많다.
{{< /callout >}}

## 4. 파이프라인 패턴 (Pipeline)

### 4.1 단계로 흐름을 만든다

**파이프라인**은 작업을 여러 **단계(stage)** 로 나누고, 데이터가 각 단계를 차례로 통과하게 만드는 작업 분해 패턴이다. 핵심은 **단계들이 겹쳐서 동시에 돈다**는 점이다 — 1번 항목이 2단계에 있을 때 2번 항목은 1단계에 들어와 있다.

공장 조립 라인이나 세탁 → 건조 → 개기의 흐름과 같다. 한정된 공유 자원(세탁기 1대, 건조기 1대)을 쉬지 않고 돌려 **전체 처리율(throughput)** 을 끌어올린다.

```
┌──────────────────────────────────────────────────────────────┐
│         파이프라인이 처리율을 높이는 원리                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  순차:  [읽기][파싱][쓰기][읽기][파싱][쓰기]                 │
│         └ 한 항목을 끝내야 다음 항목 시작 ┘                  │
│                                                              │
│  파이프라인:   시간 →                                        │
│   읽기 │[1][2][3][4]                                         │
│   파싱 │   [1][2][3][4]   ← 단계가 겹쳐 동시에 돈다          │
│   쓰기 │      [1][2][3][4]                                   │
│        └ 자원을 쉬지 않고 돌려 처리율(throughput) ↑          │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 처리율은 늘지만 지연은 그대로

파이프라인은 **처리율(단위 시간당 완료 수)** 을 높이지만, **지연(한 항목이 끝까지 가는 시간, latency)** 은 줄이지 못한다. 항목 하나는 여전히 모든 단계를 거쳐야 하기 때문이다. 또한 전체 속도는 **가장 느린 단계(병목 단계)** 에 묶인다 — 이 단계의 처리량이 파이프라인 전체의 처리량이다.

| 특성 | 파이프라인의 효과 |
|:-----|:-----------------|
| 처리율(throughput) | ↑ 증가 (단계가 겹쳐 돈다) |
| 지연(latency) | 그대로 (항목당 단계 합) |
| 한계 | 가장 느린 단계가 병목 |
| 강점 | 한정된 공유 자원으로 효율 극대화 |

### 4.3 Java로 파이프라인

각 단계를 스레드로, 단계 사이를 `BlockingQueue`로 잇는다. 큐가 단계 간 **버퍼이자 동기화 지점** 역할을 한다.

```java
// 3단계 파이프라인: 읽기 → 파싱 → 쓰기
BlockingQueue<String>  q1 = new LinkedBlockingQueue<>(64);
BlockingQueue<Record>  q2 = new LinkedBlockingQueue<>(64);

// 1단계: 읽기
Thread reader = new Thread(() -> {
    for (String line : source) q1.put(line);
    q1.put(POISON);                 // 종료 신호(sentinel)
});
// 2단계: 파싱 — 앞 큐에서 꺼내 뒤 큐로 흘려보낸다
Thread parser = new Thread(() -> {
    String line;
    while (!(line = q1.take()).equals(POISON))
        q2.put(parse(line));
    q2.put(POISON_REC);
});
// 3단계: 쓰기
Thread writer = new Thread(() -> {
    Record r;
    while ((r = q2.take()) != POISON_REC)
        write(r);
});
reader.start(); parser.start(); writer.start();
```

{{< callout type="warning" >}}
파이프라인의 균형을 맞추지 못하면 병목 단계 앞 큐가 무한정 쌓여 메모리를 잠식한다. **유계 큐(bounded queue)** 로 백프레셔(backpressure)를 걸어, 느린 단계가 앞 단계를 자연스럽게 늦추도록 해야 한다. 위 예시에서 큐 용량을 `64`로 둔 것이 그 장치다.
{{< /callout >}}

## 5. 데이터 분해 (Data Decomposition)

### 5.1 같은 '일'을 데이터 조각마다

> **데이터 분해**란 "서로 독립적으로 연산할 수 있는 단위로 데이터를 나누려면 어떻게 분해해야 하는가?"의 답을 찾는 과정이다.

해결하려는 문제에 **서로 다른 데이터 뭉치를 독립적으로 처리할 수 있는 단계**가 있다면, 데이터를 조각으로 나눠 **같은 연산을 각 조각에 동시에** 적용한다. 1억 개 원소의 합, 이미지 픽셀 변환, 대용량 로그 필터링처럼 **데이터는 큰데 연산은 단순·균질**한 문제에 잘 맞는다.

같은 명령이 여러 데이터에 적용되므로, 이는 3장 플린 분류의 **SIMD**(하나의 명령·여러 데이터)에 해당한다. GPU와 `parallelStream`이 이 모델 위에서 동작한다.

```
┌──────────────────────────────────────────────────────────────┐
│         두 갈래 분해 전략                                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  작업 분해(Task)            데이터 분해(Data)                 │
│  서로 다른 '일'을 나눔       같은 '일'을 데이터 조각마다       │
│                                                              │
│   ┌──┐┌──┐┌──┐             data ─┬─[같은 연산]─┐            │
│   │읽││계││쓰│             ──────┼─[같은 연산]─┼─ 합침       │
│   └──┘└──┘└──┘             ──────┴─[같은 연산]─┘            │
│   기능별 (MIMD/MISD)        조각별 (SIMD)                     │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 두 전략 비교

| 구분 | 작업 분해(Task) | 데이터 분해(Data) |
|:-----|:---------------|:------------------|
| 무엇을 나누나 | 기능(서로 다른 일) | 데이터(같은 일) |
| 병렬도 한계 | 기능의 개수 | 데이터 조각 수(매우 큼) |
| 플린 분류 | MIMD / MISD | SIMD |
| 적합한 문제 | 이질적 기능 조합 | 균질·대용량 데이터 |
| 대표 패턴 | 파이프라인 | 맵, 포크/조인, 맵/리듀스 |

## 6. 데이터 분해 패턴: 맵 · 포크/조인 · 맵/리듀스

데이터 분해는 몇 가지 정형화된 패턴으로 굳어져 여러 라이브러리·프레임워크가 채택하고 있다.

### 6.1 맵 (Map)

각 데이터 조각에 **같은 함수를 독립적으로** 적용해 같은 개수의 결과를 만든다. 조각 사이에 의존이 없어 가장 단순하고 병렬화가 쉽다.

```java
// 맵 — 각 원소에 같은 변환을 병렬로
List<Thumbnail> thumbs = images.parallelStream()
    .map(img -> resize(img, 128))   // 모든 조각에 동일 연산 (SIMD형)
    .toList();
```

### 6.2 포크/조인 (Fork/Join)

문제를 **재귀적으로 절반씩 쪼개(fork)** 충분히 작아지면 직접 계산하고, 결과를 거꾸로 **합치는(join)** 분할 정복 패턴이다. Java의 `ForkJoinPool`은 **작업 훔치기(work-stealing)** 로 한가한 스레드가 바쁜 스레드의 작업을 가져가 부하를 자동으로 고르게 맞춘다.

```
┌──────────────────────────────────────────────────────────────┐
│         포크/조인 — 분할 정복                                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│               [0 .. 1000]                                    │
│            fork ↙       ↘ fork                               │
│        [0..500]         [500..1000]                          │
│         ↙    ↘            ↙     ↘                            │
│   [0..250] [250..500]   ...     ...  ← 임계값까지 분할       │
│         ↘    ↙            ↘     ↙                            │
│        join(합)          join(합)                            │
│             ↘            ↙                                   │
│               join → 최종 합                                  │
└──────────────────────────────────────────────────────────────┘
```

```java
// 포크/조인 — 배열 합을 분할 정복으로
class SumTask extends RecursiveTask<Long> {
    static final int THRESHOLD = 10_000;
    final long[] a; final int lo, hi;
    SumTask(long[] a, int lo, int hi) { this.a = a; this.lo = lo; this.hi = hi; }

    protected Long compute() {
        if (hi - lo <= THRESHOLD) {          // 충분히 잘면 직접 계산
            long s = 0;
            for (int i = lo; i < hi; i++) s += a[i];
            return s;
        }
        int mid = (lo + hi) >>> 1;
        SumTask left  = new SumTask(a, lo, mid);
        left.fork();                         // 왼쪽을 풀에 맡김(비동기)
        SumTask right = new SumTask(a, mid, hi);
        long r = right.compute();            // 오른쪽은 현재 스레드가 직접
        return left.join() + r;              // 합치기(join)
    }
}
long total = new ForkJoinPool().invoke(new SumTask(data, 0, data.length));
```

### 6.3 맵/리듀스 (MapReduce)

**맵**으로 각 조각을 독립 변환한 뒤, **리듀스**로 그 결과들을 하나의 값으로 접어 모으는 패턴이다. 대규모 분산 처리(Hadoop, Spark)의 토대이며, Java 스트림에도 그대로 들어 있다.

```java
// 맵/리듀스 — 주문 금액 합계
long revenue = orders.parallelStream()  // 데이터 분해 → 병렬
    .map(Order::amount)                 // 맵 : 각 조각에 같은 추출
    .reduce(0L, Long::sum);             // 리듀스 : 결과를 하나로 접기
```

{{< callout type="warning" >}}
리듀스의 결합 함수는 **결합법칙(associativity)** 을 만족해야 한다. `(a∘b)∘c == a∘(b∘c)` 가 성립해야 조각을 어떤 순서로 합쳐도 같은 결과가 나온다. 뺄셈·나눗셈처럼 순서에 민감한 연산을 병렬 리듀스에 쓰면 결과가 달라진다. 또 맵 함수는 **부수 효과 없는 순수 함수**여야 한다 — 공유 상태를 건드리면 경쟁 조건이 생긴다(다음 장).
{{< /callout >}}

| 패턴 | 핵심 | 채택 사례 |
|:-----|:-----|:---------|
| 맵(Map) | 조각마다 같은 변환, 결과 수 동일 | `parallelStream().map` |
| 포크/조인 | 재귀 분할 → 직접 계산 → 합류 | `ForkJoinPool`, `Arrays.parallelSort` |
| 맵/리듀스 | 맵 후 결과를 하나로 접기 | Hadoop, Spark, 스트림 `reduce` |

## 7. 입도 (Granularity)

### 7.1 정의

> **입도(granularity)** 는 분해된 작업의 **수와 크기**로 결정된다. 작업이 적고 크면 **굵은 입도(coarse-grained)**, 많고 작으면 **잔 입도(fine-grained)** 다.

입도는 분해 품질을 좌우하는 핵심 손잡이다. 너무 굵으면 일부 코어가 논다(부하 불균형). 너무 잘면 작업 관리·통신 비용이 일보다 커진다(오버헤드).

```
┌──────────────────────────────────────────────────────────────┐
│         입도(granularity) 트레이드오프                        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  굵은 입도(coarse) : ███  ███  ███     작업 적고 큼          │
│    + 전환·통신 비용 ↓                                        │
│    − 부하 불균형 위험 ↑ (코어가 놀 수 있음)                  │
│                                                              │
│  잔 입도(fine)     : ▪▪▪▪▪▪▪▪▪▪▪▪     작업 많고 작음        │
│    + 부하 균형 ↑ (빈 코어를 잘 채움)                         │
│    − 작업 관리·통신 오버헤드 ↑                               │
│                                                              │
│  목표: 작업 수 ≥ 처리 자원 수, 오버헤드는 최소               │
└──────────────────────────────────────────────────────────────┘
```

### 7.2 적정 입도의 기준

이상적인 분해에서 **작업의 수는 최소한 사용 가능한 처리 자원의 수만큼은 되어야** 한다. 그래야 모든 코어가 일거리를 갖는다. 그리고 자원 수보다 **조금 더 많은 편이 유리하다** — 런타임 스케줄러가 작업을 한가한 코어로 유연하게 배치하고, 부하를 고르게 맞출 여지가 생기기 때문이다(포크/조인의 작업 훔치기가 이 여지를 활용한다).

| 입도 | 작업 수 / 크기 | 장점 | 단점 |
|:-----|:--------------|:-----|:-----|
| 굵음(coarse) | 적고 큼 | 전환·통신 비용 적음 | 부하 불균형, 코어 유휴 |
| 잠(fine) | 많고 작음 | 부하 균형, 유연성 | 관리·통신 오버헤드 |
| 적정 | 자원 수 이상, 약간 여유 | 균형 + 낮은 오버헤드 | — |

{{< callout type="info" >}}
입도는 **자원 수에 맞춰 정하되, 약간 넉넉하게** 잡는 것이 실전 지침이다. 포크/조인에서 임계값(`THRESHOLD`)을 두는 이유가 바로 이것이다 — 임계값보다 작아지면 더 쪼개지 않고 직접 계산해, 잔 입도의 오버헤드가 일을 잡아먹는 지점을 막는다. 임계값이 곧 입도 조절 손잡이다.
{{< /callout >}}

## 8. 정리

```
┌──────────────────────────────────────────────────────────────┐
│         작업 분해 한눈에 보기                                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  분해 = 문제를 동시 실행 가능한 작업으로 쪼개기               │
│   └ 만능 공식 없음 → 의존 관계 그래프(DAG)부터 그린다        │
│                                                              │
│  두 전략                                                     │
│   ├ 작업 분해 : 서로 다른 기능을 나눔 (MIMD/MISD)            │
│   │   └ 패턴: 파이프라인 (처리율 ↑)                          │
│   └ 데이터 분해: 같은 일을 조각마다 (SIMD)                   │
│       └ 패턴: 맵 · 포크/조인 · 맵/리듀스                     │
│                                                              │
│  입도(granularity) = 작업의 수·크기                          │
│   └ 자원 수 이상 + 약간 여유 (오버헤드는 최소)               │
└──────────────────────────────────────────────────────────────┘
```

### 한 줄 정리

| 개념 | 한 줄 |
|:-----|:------|
| **분해** | "문제를 동시 실행 가능한 작업으로 쪼개기 — 정답 공식은 없다" |
| **의존 관계 그래프** | "화살표 없는 작업은 동시에, 임계 경로가 한계" |
| **작업 분해** | "서로 다른 기능을 나눈다 (MIMD)" |
| **파이프라인** | "단계를 겹쳐 돌려 처리율을 높인다" |
| **데이터 분해** | "같은 일을 데이터 조각마다 동시에 (SIMD)" |
| **맵/포크조인/맵리듀스** | "변환 · 분할정복 · 접어 모으기" |
| **입도** | "자원 수 이상으로, 그러나 너무 잘게는 말 것" |

{{< callout type="info" >}}
**핵심 정리:**
1. **분해**는 동시성 프로그램의 출발점이다. 만능 공식은 없으며, **작업 의존 관계 그래프(DAG)** 를 그려 동시 실행 가능 지점과 임계 경로를 먼저 파악한다
2. **작업 분해**는 기능적으로 독립적인 작업을 나누는 전략으로 MIMD/MISD에 대응하며, 대표 패턴인 **파이프라인**은 한정된 공유 자원으로 처리율을 끌어올린다
3. **데이터 분해**는 같은 연산을 데이터 조각마다 적용하는 전략으로 SIMD에 대응하며, **맵·포크/조인·맵/리듀스** 패턴으로 정형화되어 여러 프레임워크가 채택했다
4. **입도**는 분해된 작업의 수와 크기로, 작업 수는 처리 자원 수 이상이어야 하고 약간 여유가 있으면 런타임이 부하를 유연하게 맞춘다 — 너무 잘면 오버헤드가, 너무 굵으면 부하 불균형이 생긴다
5. 실무에서는 작업 분해와 데이터 분해를 **결합**해 쓰며, 분해의 다음 단계는 나눈 작업이 공유 자원을 안전하게 다루도록 하는 **동기화**다(다음 장)
{{< /callout >}}
