단일 CPU는 한 순간에 단 하나의 명령만 실행할 수 있다. 그런데도 우리는 음악을 들으며 브라우저를 켜고 파일을 내려받는다. 어떻게 가능할까? 답은 **멀티태스킹**이다. 운영체제(런타임 시스템)가 여러 작업을 아주 빠르게 번갈아 실행해 마치 동시에 도는 것처럼 보이게 만든다. 이 장에서는 멀티태스킹의 동작 원리, 그 핵심 장치인 **컨텍스트 스위칭**과 **스케줄러**, 작업을 나누는 기준인 **CPU 바운드 vs I/O 바운드**, 그리고 **협력형 / 선점형** 두 방식을 살펴본다.

---

## 1. 멀티태스킹이란?

### 1.1 정의

> **멀티태스킹**은 일정 시간 동안 여러 작업을 동시에 교차 실행하는 것을 말한다. 멀티태스킹은 **런타임 시스템 수준의 기능**이며, 하드웨어 수준에는 존재하지 않는 개념이다.

여러 작업을 동시에 수행하는 능력은 런타임 시스템에 필수 불가결하다. 멀티태스킹은 작업의 **동시 교차 실행**과 **실행 전환**을 제어한다. 지속적인 교차 실행 덕분에, 물리적 병렬 실행이 아니어도 여러 작업이 동시에 도는 듯한 **착각(illusion of concurrency)** 을 일으킬 수 있다.

### 1.2 동시성의 착각

```
┌──────────────────────────────────────────────────────────────┐
│         단일 CPU에서의 멀티태스킹 (시간 교차)                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  시간 →                                                      │
│  CPU │ T1 │ T2 │ T3 │ T1 │ T2 │ T3 │ ...  ← 빠르게 전환      │
│                                                              │
│  사용자 눈에는: T1·T2·T3 이 "동시에" 도는 것처럼 보임          │
│  실제로는:     한 순간에 오직 하나의 작업만 CPU 점유           │
│                                                              │
│  ── 병렬 실행(멀티코어)과의 차이 ──                            │
│  CPU0 │ T1 T1 T1 T1 │   ← 진짜 동시 (물리적 병렬)            │
│  CPU1 │ T2 T2 T2 T2 │                                        │
└──────────────────────────────────────────────────────────────┘
```

{{< callout type="info" >}}
**멀티태스킹 ≠ 병렬성.** 멀티태스킹은 *하나의* 처리 장치에서 작업을 빠르게 번갈아 실행하는 소프트웨어 기법이고, 병렬성은 *여러* 처리 장치에서 작업을 물리적으로 동시에 실행하는 것이다. 멀티코어에서는 둘이 결합된다 — 코어마다 병렬로 돌면서, 각 코어가 다시 멀티태스킹으로 여러 작업을 교차 실행한다. (2장 *동시성 vs 병렬성* 참고)
{{< /callout >}}

## 2. 작업의 두 유형: CPU 바운드 vs I/O 바운드

### 2.1 병목 지점에 따른 분류

프로그램의 병목 지점은 집중적으로 사용되는 자원의 유형에 따라 둘로 나뉜다. 이 구분은 멀티태스킹을 이해하는 출발점이다 — 스케줄러가 작업을 어떻게 다룰지, 스레드 풀을 얼마나 크게 잡을지가 여기서 갈린다.

| 구분 | CPU 바운드 | I/O 바운드 |
|:-----|:----------|:----------|
| 주 사용 자원 | 프로세서(연산 능력) | 입출력 장치 |
| 속도 제한 요소 | 시스템의 연산 능력 | 하드웨어 I/O 속도 |
| 대기 시간 | 거의 없음 (CPU 풀가동) | 대부분 대기 |
| 예시 | 암호화, 영상 인코딩, 행렬 곱, 압축 | 디스크 읽기, DB 조회, 외부 API 호출 |

```
┌──────────────────────────────────────────────────────────────┐
│         두 유형의 CPU 점유 패턴                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  CPU 바운드:  ████████████████████████  (CPU 계속 점유)      │
│                                                              │
│  I/O 바운드:  ██····대기····██····대기····██  (대부분 대기)   │
│               ↑계산  ↑디스크/네트워크 응답 기다림              │
│                                                              │
│  → I/O 바운드는 "대기" 동안 CPU가 놀므로,                     │
│    그 틈에 다른 작업을 실행하면 효율이 올라간다                │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 왜 이 구분이 중요한가

I/O 바운드 작업은 대기하는 동안 CPU를 쓰지 않는다. 멀티태스킹은 바로 이 빈 시간을 다른 작업으로 채워 처리량을 끌어올린다. 반대로 CPU 바운드 작업은 코어를 꽉 채워 돌리므로, 코어 수보다 많은 작업을 동시에 돌려봐야 컨텍스트 스위칭 비용만 늘어난다.

### 2.3 Java로 구분해 보기

```java
// CPU 바운드 — 연산이 CPU를 계속 점유한다
long countPrimes(int limit) {
    return java.util.stream.IntStream.rangeClosed(2, limit)
        .filter(MultitaskingDemo::isPrime)   // 순수 계산
        .count();
}

static boolean isPrime(int n) {
    for (int i = 2; (long) i * i <= n; i++)
        if (n % i == 0) return false;
    return true;
}

// I/O 바운드 — 응답을 기다리며 CPU는 대부분 논다
String fetch(String url) throws Exception {
    var client = java.net.http.HttpClient.newHttpClient();
    var req = java.net.http.HttpRequest.newBuilder()
        .uri(java.net.URI.create(url)).build();
    // send() 호출 동안 스레드는 네트워크 응답을 '대기'한다
    return client.send(req,
        java.net.http.HttpResponse.BodyHandlers.ofString()).body();
}
```

{{< callout type="info" >}}
풀 크기 산정 공식이 두 유형에서 달랐던 이유가 여기에 있다(5장 *스레드 풀*). CPU 바운드는 `코어 수` 근처가 최적이고, I/O 바운드는 대기 시간을 메우기 위해 `코어 수 × (1 + 대기/연산)` 만큼 늘린다.
{{< /callout >}}

## 3. 컨텍스트 스위칭

### 3.1 정의

> **컨텍스트 스위칭**은 한 작업의 컨텍스트를 나중에 재개할 수 있도록 저장하고, 다른 작업의 컨텍스트로 전환하는 물리적 행위다. 주로 운영체제가 수행하며, 운영체제가 제공하는 멀티태스킹 기능의 핵심 요소다.

작업을 전환하려면 "지금까지 어디까지 했는지"를 통째로 보관했다가 나중에 그대로 복원해야 한다. 이 **컨텍스트(상태 묶음)** 에는 다음이 포함된다.

| 저장 대상 | 설명 |
|:---------|:-----|
| 프로그램 카운터(PC) | 다음에 실행할 명령의 주소 |
| 레지스터 | 연산 중이던 CPU 레지스터 값들 |
| 스택 포인터 | 호출 스택의 현재 위치 |
| 메모리 맵 / 페이지 테이블 | (프로세스 전환 시) 주소 공간 정보 |

### 3.2 동작 흐름

```
┌──────────────────────────────────────────────────────────────┐
│         컨텍스트 스위칭의 단계                                 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  작업 A 실행 중                                               │
│       │  ① 인터럽트/타이머 발생                               │
│       ▼                                                      │
│  A의 컨텍스트 저장  (PC·레지스터·스택 → PCB/TCB)              │
│       │                                                      │
│       ▼                                                      │
│  스케줄러가 다음 작업 B 선택                                   │
│       │                                                      │
│       ▼                                                      │
│  B의 컨텍스트 복원  (저장해 둔 상태를 CPU에 적재)             │
│       │                                                      │
│       ▼                                                      │
│  작업 B 실행 재개                                             │
│                                                              │
│  ※ 전환 그 자체는 '일'을 하지 않는 순수 오버헤드               │
└──────────────────────────────────────────────────────────────┘
```

### 3.3 비용: 직접 비용과 간접 비용

컨텍스트 스위칭은 비교적 비용이 많이 든다. 비용은 두 종류다.

- **직접 비용**: 레지스터·PC·스택을 저장하고 복원하는 시간.
- **간접 비용**: 전환 직후 **CPU 캐시와 TLB가 식는다(cache/TLB pollution)**. 새 작업은 캐시에 데이터가 없어 메모리에서 다시 읽어야 하므로, 한동안 느리게 동작한다. 이 간접 비용이 종종 직접 비용보다 크다.

{{< callout type="warning" >}}
컨텍스트 스위칭은 순수 오버헤드다. 애플리케이션 내 작업(스레드) 수를 주의해서 관리해야 한다. 작업 수가 지나치게 많으면 **실제 일보다 전환에 쓰는 시간이 늘어나** 전체 성능이 오히려 떨어진다. 이 현상이 극단으로 가면 시스템이 전환만 반복하며 진척이 없는 **스래싱(thrashing)** 에 빠진다.
{{< /callout >}}

### 3.4 프로세스 전환 vs 스레드 전환

```
┌──────────────────────────────────────────────────────────────┐
│         전환 비용 비교                                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  프로세스 ↔ 프로세스                                          │
│   - 주소 공간(페이지 테이블) 교체                             │
│   - TLB 전체 무효화 → 캐시 효과 큼                            │
│   - 비용 큼                                                   │
│                                                              │
│  스레드 ↔ 스레드 (같은 프로세스 내)                           │
│   - 주소 공간 공유 → 페이지 테이블 그대로                     │
│   - 레지스터·스택만 교체                                      │
│   - 비용 작음                                                 │
└──────────────────────────────────────────────────────────────┘
```

같은 프로세스 안의 스레드끼리는 주소 공간을 공유하므로 페이지 테이블을 갈아끼울 필요가 없다. 그래서 **스레드 전환이 프로세스 전환보다 훨씬 싸다.** 이것이 다음 절(6장)에서 "단일 프로세스 다중 스레드가 더 효율적"이라고 말하는 이유 중 하나다.

## 4. 멀티태스킹의 두 방식: 협력형 vs 선점형

작업 사이의 전환 시점을 **누가 결정하느냐**에 따라 멀티태스킹은 두 갈래로 나뉜다.

### 4.1 협력형 멀티태스킹 (Cooperative)

각 작업이 **자발적으로** 제어권을 양보(yield)할 때만 전환이 일어난다. 작업이 "이제 잠깐 다른 작업 하세요" 하고 스스로 비켜주는 방식이다.

- 전환 지점이 명확해 동기화가 단순하고, 전환 비용이 적다.
- 치명적 약점: **한 작업이 양보하지 않으면 시스템 전체가 멈춘다.** 무한 루프나 긴 계산 하나가 모든 작업을 인질로 잡는다.
- 과거 사례: Windows 3.x, 고전 Mac OS.
- 현대의 부활: **코루틴, async/await, 제너레이터, 이벤트 루프**(Node.js, Python `asyncio`)가 모두 협력형 모델이다. `await` 지점이 곧 자발적 양보 지점이다.

### 4.2 선점형 멀티태스킹 (Preemptive)

> **선점형 멀티태스킹**에서는 스케줄러가 작업의 우선순위를 매기며, 제어권을 강제로 빼앗아 다른 작업에 넘길 수 있다.

스케줄러가 타이머 인터럽트를 기준으로 정해진 시간(**타임 슬라이스**)이 지나면 작업을 강제로 중단시키고 다른 작업으로 전환한다.

- 한 작업이 CPU를 독점할 수 없으므로 **응답성과 공정성**이 보장된다.
- 전환이 임의 시점에 끼어들 수 있어 **공유 데이터 동기화가 필수**다(4장 *스레드 위험성*).
- 현대 운영체제(Linux, Windows, macOS)의 표준 방식이다.

### 4.3 비교

| 구분 | 협력형(Cooperative) | 선점형(Preemptive) |
|:-----|:-------------------|:-------------------|
| 전환 결정 | 작업이 자발적으로 양보 | 스케줄러가 강제 전환 |
| 전환 시점 | 명시적 `yield`/`await` | 타임 슬라이스 만료, 인터럽트 |
| 독점 위험 | 있음 (한 작업이 멈추면 전체 정지) | 없음 |
| 동기화 부담 | 낮음 (전환 지점이 명확) | 높음 (임의 시점 전환) |
| 전환 비용 | 낮음 | 상대적으로 높음 |
| 사례 | 코루틴, async/await, 이벤트 루프 | 현대 OS 스레드 스케줄링 |

```
┌──────────────────────────────────────────────────────────────┐
│         두 방식의 전환 시점                                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  협력형:   T1 ███████[yield] T2 ████[yield] T1 ██████         │
│            └ 작업이 스스로 양보하는 순간에만 전환 ┘            │
│                                                              │
│  선점형:   T1 ███│ T2 ███│ T1 ███│ T2 ███│  (│=타임 슬라이스) │
│            └ 정해진 시간이 지나면 강제로 전환 ┘                │
└──────────────────────────────────────────────────────────────┘
```

{{< callout type="info" >}}
**협력형의 부활.** 선점형이 표준이 된 지 오래지만, 고성능 I/O 영역에서 협력형이 다시 떠올랐다. 수십만 개의 연결을 다루는 서버에서 OS 스레드를 그만큼 만들면 컨텍스트 스위칭 비용이 폭발한다. 그래서 `asyncio`·코루틴·Java 21 **가상 스레드**는 `await`/블로킹 지점에서만 협력적으로 양보하며, 적은 수의 OS 스레드 위에서 수많은 논리 작업을 돌린다. (5장 *가상 스레드* 참고)
{{< /callout >}}

## 5. 스케줄러

### 5.1 역할

스케줄러는 **어떤 작업을 언제, 얼마나 실행할지** 결정하는 운영체제의 핵심 구성 요소다. 준비된 작업들 중 하나를 골라 CPU에 올리고, 타임 슬라이스가 끝나거나 작업이 대기 상태로 가면 다음 작업을 고른다.

```
┌──────────────────────────────────────────────────────────────┐
│         스케줄러와 작업 상태                                   │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌─────────┐  스케줄러 선택   ┌─────────┐                   │
│   │  준비    │ ───────────────→ │  실행    │                  │
│   │ (Ready) │ ←─────────────── │(Running)│                  │
│   └─────────┘  타임 슬라이스    └────┬────┘                  │
│        ▲          만료(선점)         │ I/O 요청              │
│        │                             ▼                       │
│        │   I/O 완료            ┌─────────┐                   │
│        └──────────────────────│  대기    │                   │
│                                │(Blocked)│                   │
│                                └─────────┘                   │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 타임 슬라이스(타임 퀀텀)

선점형 스케줄러는 각 작업에 **타임 슬라이스**(time slice, quantum)라는 실행 시간 조각을 할당한다. 이 값이 핵심 트레이드오프를 만든다.

- **너무 짧으면**: 전환이 잦아져 컨텍스트 스위칭 오버헤드가 커진다.
- **너무 길면**: 응답성이 떨어진다(다른 작업이 오래 기다린다).

### 5.3 대표 스케줄링 알고리즘

| 알고리즘 | 핵심 | 특징 |
|:--------|:-----|:-----|
| FCFS | 먼저 온 작업 먼저 | 단순하지만 긴 작업이 뒤를 막음(convoy) |
| Round-Robin | 타임 슬라이스 순환 | 공정함, 응답성 좋음 |
| Priority | 우선순위 높은 작업 먼저 | 기아(starvation) 위험 |
| MLFQ | 다단계 피드백 큐 | I/O 바운드 우대, 실무 OS가 채택 |

### 5.4 CPU/I/O 바운드를 구분하는 스케줄러

> 시스템 자원을 효율적으로 활용하려면 스케줄러가 CPU 중심 연산과 입출력 중심 연산을 구분할 수 있어야 한다.

영리한 스케줄러는 **I/O 바운드 작업의 우선순위를 높인다.** I/O 바운드 작업은 잠깐 CPU를 쓰고 곧 대기로 빠지므로, 먼저 실행시켜 I/O 요청을 던져두면 그 대기 시간 동안 CPU 바운드 작업이 코어를 채울 수 있다. 결과적으로 CPU와 I/O 장치가 동시에 바쁘게 돌아간다. 리눅스의 CFS, Windows의 스케줄러가 이런 휴리스틱을 사용한다.

## 6. 프로세스 vs 스레드로 멀티태스킹

멀티태스킹의 단위로 **프로세스**를 쓸 수도, **스레드**를 쓸 수도 있다.

### 6.1 왜 스레드가 더 효율적인가

> 다중 프로세스보다는 단일 프로세스 다중 스레드 형태가 일반적으로 더 효율적이다.

| 비교 | 다중 프로세스 | 단일 프로세스 · 다중 스레드 |
|:-----|:-------------|:--------------------------|
| 메모리 | 각자 독립 주소 공간(중복) | 주소 공간 공유(절약) |
| 생성 비용 | 큼 | 작음 |
| 컨텍스트 스위칭 | 비쌈(페이지 테이블 교체) | 쌈(레지스터·스택만) |
| 데이터 공유 | IPC 필요(5장) | 메모리 직접 공유 |
| 격리·안정성 | 강함(하나 죽어도 무관) | 약함(하나가 전체를 죽일 수 있음) |

```
┌──────────────────────────────────────────────────────────────┐
│         두 멀티태스킹 단위                                     │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   다중 프로세스               단일 프로세스 · 다중 스레드       │
│   ┌────────┐ ┌────────┐      ┌──────────────────────┐        │
│   │ PROC A │ │ PROC B │      │      PROC             │        │
│   │ 주소공간│ │ 주소공간│      │  공유 주소 공간        │        │
│   │  스레드 │ │  스레드 │      │ ┌────┐┌────┐┌────┐   │        │
│   └────────┘ └────────┘      │ │ T1 ││ T2 ││ T3 │   │        │
│    격리 ↑ 비용 ↑              │ └────┘└────┘└────┘   │        │
│                              │  공유 ↑ 비용 ↓        │        │
│                              └──────────────────────┘        │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 트레이드오프

스레드가 싸고 빠르다고 항상 정답은 아니다. 주소 공간을 공유하므로 **한 스레드의 오류(메모리 손상, 미처리 예외)가 프로세스 전체를 무너뜨릴 수 있다.** 반면 프로세스는 격리되어 하나가 죽어도 나머지가 멀쩡하다. 그래서 안정성·보안이 중요한 경우(크롬의 탭별 프로세스 격리 등)에는 의도적으로 다중 프로세스를 택한다.

{{< callout type="warning" >}}
스레드의 효율성은 **공유 가변 데이터**라는 대가를 동반한다. 여러 스레드가 같은 메모리를 동시에 건드리면 경쟁 조건이 발생한다(4장). 즉, "스레드가 더 효율적"이라는 명제는 *동기화를 올바르게 했을 때*라는 전제 위에서만 성립한다.
{{< /callout >}}

## 7. 멀티태스킹과 현대 런타임

멀티태스킹 개념은 운영체제 안에만 머물지 않는다. 런타임과 프레임워크가 같은 아이디어를 사용자 공간에서 재현한다.

| 런타임 | 모델 | 핵심 |
|:------|:-----|:-----|
| Node.js | 단일 스레드 이벤트 루프 | 협력형, 콜백/`await`로 양보 |
| Python `asyncio` | 코루틴 + 이벤트 루프 | 협력형, `await`가 양보 지점 |
| Go | 고루틴 + 런타임 스케줄러 | M:N, 일부 선점 |
| Java 21 | 가상 스레드 | 블로킹 지점에서 협력적 양보 |

```java
// Java 21 — 가상 스레드로 수만 개의 I/O 작업을 멀티태스킹
try (var exec = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100_000; i++) {
        exec.submit(() -> {
            // 블로킹 I/O 지점에서 가상 스레드가 '협력적으로' 양보 →
            // 소수의 OS 스레드(캐리어) 위에서 수만 작업이 교차 실행된다
            return fetch("https://example.com");
        });
    }
}
```

> 핵심 통찰: **이벤트 루프·코루틴·가상 스레드는 모두 "협력형 멀티태스킹"의 현대적 재해석**이다. OS 스레드라는 비싼 자원을 적게 쓰면서, 그 위에서 수많은 논리 작업을 교차 실행한다.

## 8. 정리

```
┌──────────────────────────────────────────────────────────────┐
│         멀티태스킹 한눈에 보기                                 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  멀티태스킹 = 여러 작업을 빠르게 교차 실행 (런타임 기능)        │
│   └ 단일 CPU에서도 '동시성의 착각'을 만든다                    │
│                                                              │
│  작업 유형                                                    │
│   ├ CPU 바운드 : 연산이 병목 (코어 수만큼)                    │
│   └ I/O 바운드 : 대기가 병목 (대기 틈을 메움)                 │
│                                                              │
│  컨텍스트 스위칭 = 작업 상태 저장·복원 (순수 오버헤드)         │
│   └ 직접 비용 + 캐시/TLB 오염(간접 비용)                      │
│                                                              │
│  두 방식                                                      │
│   ├ 협력형 : 작업이 자발적 양보 (코루틴, async/await)         │
│   └ 선점형 : 스케줄러가 강제 전환 (현대 OS 표준)              │
│                                                              │
│  단위 : 스레드(싸고 공유) vs 프로세스(비싸지만 격리)           │
└──────────────────────────────────────────────────────────────┘
```

### 한 줄 정리

| 개념 | 한 줄 |
|:-----|:------|
| **멀티태스킹** | "하나의 CPU로 여러 작업을 번갈아 — 동시인 척한다" |
| **CPU 바운드** | "계산이 병목, 코어를 꽉 채운다" |
| **I/O 바운드** | "대기가 병목, 그 틈에 다른 일을 시킨다" |
| **컨텍스트 스위칭** | "상태를 저장했다 복원 — 전환 자체는 공짜가 아니다" |
| **협력형** | "작업이 스스로 비켜줄 때만 전환" |
| **선점형** | "스케줄러가 시간 다 되면 강제로 빼앗는다" |

{{< callout type="info" >}}
**핵심 정리:**
1. **멀티태스킹**은 하드웨어가 아닌 **런타임 시스템 수준**의 기능으로, 작업을 빠르게 교차 실행해 동시성의 착각을 만든다
2. 작업은 병목에 따라 **CPU 바운드 / I/O 바운드**로 나뉘며, 이 구분이 스케줄링과 스레드 풀 크기를 좌우한다
3. **컨텍스트 스위칭**은 작업 상태를 저장·복원하는 순수 오버헤드이며, 간접 비용(캐시·TLB 오염)이 크다 — 작업 수를 무분별하게 늘리면 안 된다
4. 전환을 누가 결정하느냐에 따라 **협력형(자발적 양보)** 과 **선점형(강제 전환)** 으로 나뉜다. 현대 OS는 선점형이 표준이지만, 고성능 I/O에서는 협력형(코루틴·가상 스레드)이 부활했다
5. 멀티태스킹 단위로 **스레드**는 싸고 공유에 유리하지만 격리가 약하고, **프로세스**는 비싸지만 안정적이다 — 동기화를 전제로 스레드가 효율적이다
{{< /callout >}}
