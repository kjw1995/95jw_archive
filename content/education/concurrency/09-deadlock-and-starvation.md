동기화(8장)는 경쟁 조건을 막아 주지만, 락을 부주의하게 쓰면 이번엔 **작업이 멈춰 버리는** 새로운 문제를 부른다. 이 장에서는 서로가 상대의 자원을 기다리며 영원히 멈추는 **교착 상태(deadlock)**, 멈추지는 않지만 진행도 못 하는 **라이브락(livelock)**, 그리고 자원을 아예 할당받지 못하는 **기아 상태(starvation)** 를 살펴본다. 마지막으로 이런 문제들이 반복해서 나타나는 **고전적 동시성 문제**(프로듀서-컨슈머, 리더-라이터)와 그 정석 해법을 짚는다.

---

## 1. 동기화가 부르는 새로운 문제

경쟁 조건을 막으려고 락을 걸기 시작하면, 이번엔 **락 자체가 만드는 정체(停滯)** 가 문제가 된다. 증상은 셋으로 나뉜다.

```
┌──────────────────────────────────────────────────────────────┐
│         멈춤의 세 가지 얼굴                                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  교착 상태   서로 상대의 자원을 기다리며 완전히 멈춤          │
│  (deadlock)  → 아무도 진행 못 함, 저절로 안 풀림             │
│                                                              │
│  라이브락    계속 양보·재시도하지만 진행이 없음              │
│  (livelock)  → 바쁘게 움직이나 일이 안 됨                    │
│                                                              │
│  기아 상태   욕심 많은 작업에 밀려 자원을 못 받음            │
│  (starvation)→ 남들은 도는데 특정 작업만 굶음               │
└──────────────────────────────────────────────────────────────┘
```

세 문제 모두 **작업에 맡긴 일이 끝나지 않는다**는 공통점이 있다. 경쟁 조건이 "틀린 값"을 낸다면, 이 문제들은 "값 자체가 나오지 않는" 활성(liveness) 실패다.

{{< callout type="info" >}}
동시성 버그는 크게 **안전성(safety) 문제**와 **활성(liveness) 문제**로 나뉜다. 8장의 경쟁 조건은 "나쁜 일이 절대 일어나지 않아야 한다"는 안전성 위반이고, 이 장의 교착·라이브락·기아는 "좋은 일이 결국 일어나야 한다"는 활성 위반이다. 둘은 자주 **상충**한다 — 안전을 위해 락을 더 걸수록 활성은 위험해진다.
{{< /callout >}}

## 2. 교착 상태

### 2.1 정의

> **교착 상태(deadlock)** 란 둘 이상의 작업이 각자 상대가 쥔 자원이 해제되기를 기다리며, **서로 영원히 멈춰 서는** 상황이다.

가장 흔한 형태는 **락을 서로 엇갈린 순서로** 잡을 때 생긴다. A는 락1을 쥐고 락2를 기다리고, B는 락2를 쥐고 락1을 기다린다. 둘 다 상대가 먼저 놓아 주기를 기다리므로 누구도 진행하지 못한다.

```
┌──────────────────────────────────────────────────────────────┐
│         엇갈린 락 순서가 만드는 교착                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   스레드 A            스레드 B                                │
│   락1 획득 ✔         락2 획득 ✔                              │
│      │                  │                                    │
│   락2 요청 … 대기     락1 요청 … 대기                        │
│      │                  │                                    │
│      └──── B가 쥔 락2 기다림                                 │
│                         └──── A가 쥔 락1 기다림              │
│                                                              │
│   서로 상대를 기다림 → 영원히 멈춤 (순환 대기)               │
└──────────────────────────────────────────────────────────────┘
```

```java
// 교착 상태를 만드는 코드 — 두 스레드가 락을 엇갈려 잡는다
Object lock1 = new Object();
Object lock2 = new Object();

// 스레드 A
synchronized (lock1) {
    synchronized (lock2) { transfer(); }   // lock1 → lock2 순서
}

// 스레드 B
synchronized (lock2) {
    synchronized (lock1) { transfer(); }   // lock2 → lock1 순서 (역순!)
}
```

### 2.2 교착 상태의 네 가지 조건

교착 상태는 아무 때나 생기지 않는다. 다음 **네 조건이 모두 동시에** 성립할 때만 발생한다(코프먼 조건, Coffman conditions). 뒤집으면, **하나만 깨도** 교착을 막을 수 있다는 뜻이다.

| 조건 | 뜻 | 깨는 방법 |
|:-----|:-----|:-----|
| 상호 배제 | 자원을 한 번에 한 작업만 점유 | 자원을 공유 가능하게(읽기 전용 등) |
| 점유와 대기 | 자원을 쥔 채 다른 자원을 기다림 | 필요한 락을 **한꺼번에** 획득 |
| 비선점 | 남이 쥔 자원을 강제로 뺏지 못함 | 타임아웃 후 자진 해제(back off) |
| 순환 대기 | 대기 관계가 원을 이룸 | **락 획득 순서를 전역에서 통일** |

{{< callout type="warning" >}}
실무에서 가장 현실적인 예방책은 **순환 대기 깨기** — 즉 **모든 스레드가 락을 같은 순서로 획득**하게 강제하는 것이다. 위 예제에서 A와 B가 둘 다 `lock1 → lock2` 순으로만 잡으면 교착은 절대 생기지 않는다. 락에 전역 순번(예: 계좌 ID 크기순)을 매기고 항상 오름차순으로 잡는 것이 정석이다.
{{< /callout >}}

### 2.3 예방·회피·탐지

교착 상태에 대한 대응은 크게 세 갈래다.

```
┌──────────────────────────────────────────────────────────────┐
│         교착 상태 대응 전략                                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  예방  네 조건 중 하나를 원천 차단                            │
│        └ 락 순서 통일, 락 한꺼번에 잡기 (가장 실용적)        │
│                                                              │
│  회피  실행 중 안전 상태만 유지 (은행원 알고리즘)            │
│        └ 자원 요청을 미리 검사, 구현·비용 큼                 │
│                                                              │
│  탐지·복구  일단 두고 주기적으로 발견 → 강제 해제            │
│        └ DB가 교착 감지 후 한쪽 트랜잭션 롤백               │
└──────────────────────────────────────────────────────────────┘
```

```java
// 타임아웃으로 비선점 조건 깨기 — 일정 시간 못 얻으면 물러난다
if (lock1.tryLock(1, TimeUnit.SECONDS)) {
    try {
        if (lock2.tryLock(1, TimeUnit.SECONDS)) {
            try { transfer(); }
            finally { lock2.unlock(); }
        }
        // 락2 못 얻으면 락1도 풀고 재시도 → 교착 대신 재시도
    } finally { lock1.unlock(); }
}
```

{{< callout type="info" >}}
관계형 DB는 대부분 **탐지·복구** 방식을 쓴다. 두 트랜잭션이 서로의 행 락을 기다리면 DB가 대기 그래프에서 순환을 발견하고, 한쪽을 **교착 상태 희생자(deadlock victim)** 로 골라 롤백시킨다(예: PostgreSQL의 `deadlock detected`, MySQL InnoDB의 자동 감지). 애플리케이션은 이 예외를 잡아 **재시도**하도록 설계해야 한다.
{{< /callout >}}

## 3. 라이브락

> **라이브락(livelock)** 은 작업들이 멈추지는 않지만, 서로의 동작에 계속 반응해 상태만 바꿀 뿐 **실질적인 진행이 없는** 상황이다.

교착 상태는 완전히 멈춰 있어 오히려 알아채기 쉽다. 라이브락은 스레드가 **바쁘게 CPU를 쓰며** 락을 잡았다 놓았다 반복하므로, 겉보기엔 살아 있어 더 헷갈린다.

```
┌──────────────────────────────────────────────────────────────┐
│         라이브락 — 좁은 복도의 두 사람                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   A →        ← B     서로 마주침                             │
│   A ← (양보)  B →     동시에 같은 쪽으로 비킴                 │
│   A →   (양보) ← B    또 같은 쪽으로 비킴                     │
│   … 계속 서로 양보만, 아무도 못 지나감                        │
│                                                              │
│   움직임은 있으나 진행은 0                                    │
└──────────────────────────────────────────────────────────────┘
```

두 스레드가 교착을 피하려고 "락을 못 잡으면 물러난다"를 넣었는데, 둘이 **완벽히 같은 타이밍**으로 물러나고 다시 시도하면 이 짓을 무한 반복한다. 해법은 물러나는 시점에 **무작위 지연(random back-off)** 을 주어 타이밍을 어긋나게 하는 것이다 — 네트워크의 CSMA/CD가 충돌 후 무작위 대기하는 것과 같은 원리다.

## 4. 기아 상태

> **기아 상태(starvation)** 는 한 작업이 다른 '욕심 많은' 작업들에 계속 밀려 **CPU 시간이나 공유 자원을 아예 할당받지 못하는** 상황이다.

교착·라이브락이 **여러 작업의 관계**에서 생긴다면, 기아는 **한 작업이 소외**되는 문제다. 원인은 대개 스케줄링과 동기화 정책에 있다.

```
┌──────────────────────────────────────────────────────────────┐
│         기아 상태 — 우선순위에 밀리는 작업                     │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   고우선 작업들 ──▶ 자원 계속 차지                            │
│   고우선 작업들 ──▶ 자원 계속 차지                            │
│        …                                                     │
│   저우선 작업   ──▶ 대기 … 대기 … 대기 …  (영원히 굶음)      │
│                                                              │
│   원인: 우선순위 편중, 불공정 락, 잘못된 스케줄링             │
└──────────────────────────────────────────────────────────────┘
```

| 기아를 부르는 원인 | 설명 |
|:-----|:-----|
| 우선순위 편중 | 높은 우선순위 작업이 CPU를 독차지 |
| 불공정 락 | 먼저 기다린 스레드를 건너뛰고 다른 스레드에 락 부여 |
| 잘못된 스케줄링 | 특정 작업을 계속 뒤로 미루는 알고리즘 |

{{< callout type="info" >}}
기아의 대표적 대책은 **공정성(fairness)** 을 도입하는 것이다. `new ReentrantLock(true)` 처럼 **공정 모드** 락은 대기 큐의 **가장 오래 기다린 스레드에게 먼저** 락을 준다(FIFO). 다만 공정성은 처리율을 떨어뜨리는 비용이 있어, 기아가 실제로 문제될 때만 쓴다. 스케줄러 차원에서는 오래 기다린 작업의 우선순위를 점점 올리는 **에이징(aging)** 기법을 쓴다.
{{< /callout >}}

### 우선순위 역전

기아와 얽힌 유명한 함정이 **우선순위 역전(priority inversion)** 이다. 낮은 우선순위 작업이 락을 쥔 채 높은 우선순위 작업을 막고, 그 사이 중간 우선순위 작업이 낮은 작업의 CPU를 빼앗으면, 결과적으로 **높은 우선순위 작업이 중간 작업에 밀리는** 뒤집힘이 생긴다. 1997년 화성 탐사선 패스파인더가 이 문제로 재부팅을 반복한 사례가 유명하다. 해법은 락을 쥔 낮은 작업의 우선순위를 일시적으로 끌어올리는 **우선순위 상속(priority inheritance)** 이다.

## 5. 고전적 동시성 문제 ① 프로듀서-컨슈머

동시성은 오래된 주제라, 반복되는 문제들엔 이미 **모범 사례(디자인 패턴)** 가 정리돼 있다. 그중 가장 유명한 둘을 뮤텍스·세마포어(8장)로 푼다.

> **프로듀서-컨슈머 문제**: 데이터를 **생산**하는 작업과 **소비**하는 작업이 **유한한 버퍼**를 공유할 때, 버퍼가 가득 찼는데 넣거나 비었는데 꺼내는 오류 없이 조율하는 문제.

```
┌──────────────────────────────────────────────────────────────┐
│         프로듀서-컨슈머 — 유한 버퍼 공유                       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   프로듀서 ──put──▶ [ □ □ ■ ■ □ ] ──take──▶ 컨슈머          │
│                     유한 버퍼(크기 N)                         │
│                                                              │
│   버퍼 가득참 → 프로듀서 대기                                 │
│   버퍼 비었음 → 컨슈머 대기                                   │
│                                                              │
│   세마포어 2개로 빈칸/찬칸 개수를 신호                        │
└──────────────────────────────────────────────────────────────┘
```

핵심은 **세마포어 두 개**로 "빈 칸 수"와 "찬 칸 수"를 세는 것이다. 프로듀서는 빈 칸 허가증을 얻어야 넣고, 컨슈머는 찬 칸 허가증을 얻어야 꺼낸다. 버퍼 자체의 상호 배제는 뮤텍스가 맡는다.

```java
// 세마포어 2개 + 뮤텍스로 유한 버퍼 조율
Semaphore empty = new Semaphore(N);   // 빈 칸 수 (처음 N)
Semaphore full  = new Semaphore(0);   // 찬 칸 수 (처음 0)
Lock mutex = new ReentrantLock();

// 프로듀서
empty.acquire();                 // 빈 칸 있어야 진행 (없으면 대기)
mutex.lock();
try { buffer.add(item); }
finally { mutex.unlock(); }
full.release();                  // 찬 칸 하나 늘었음을 알림

// 컨슈머
full.acquire();                  // 찬 칸 있어야 진행 (없으면 대기)
mutex.lock();
try { item = buffer.remove(); }
finally { mutex.unlock(); }
empty.release();                 // 빈 칸 하나 늘었음을 알림
```

{{< callout type="info" >}}
자바에서는 이 패턴을 직접 짤 필요가 거의 없다. `BlockingQueue`(예: `ArrayBlockingQueue`)가 프로듀서-컨슈머를 그대로 구현해, `put()`은 가득 차면 자동으로 막히고 `take()`는 비면 자동으로 막힌다. 스레드 풀(5장)의 작업 큐도 내부적으로 이 구조다.
{{< /callout >}}

## 6. 고전적 동시성 문제 ② 리더-라이터

> **리더-라이터 문제**: 여러 작업이 공유 데이터를 **읽거나(read) 쓸(write)** 때, **읽기는 여럿이 동시에** 허용하되 **쓰기는 홀로**만 하도록 조율하는 문제.

읽기끼리는 서로 값을 바꾸지 않으므로 동시에 해도 안전하다. 문제는 쓰기다 — 쓰는 동안엔 아무도(읽는 이조차) 접근하면 안 된다. 뮤텍스로 전부 막으면 안전하지만 읽기 동시성이 사라져 낭비다.

```
┌──────────────────────────────────────────────────────────────┐
│         리더-라이터 — 읽기는 공유, 쓰기는 독점                 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   읽기 락:  R R R   ← 여러 리더 동시 허용                    │
│   쓰기 락:    W     ← 라이터 홀로, 그동안 R·W 모두 차단      │
│                                                              │
│   읽기 다수 + 쓰기 드묾 → 처리율 크게 향상                    │
└──────────────────────────────────────────────────────────────┘
```

```java
// ReadWriteLock — 읽기 공유, 쓰기 배타
ReadWriteLock rw = new ReentrantReadWriteLock();

// 리더 (여럿 동시 가능)
rw.readLock().lock();
try { return cache.get(key); }
finally { rw.readLock().unlock(); }

// 라이터 (홀로)
rw.writeLock().lock();
try { cache.put(key, value); }
finally { rw.writeLock().unlock(); }
```

{{< callout type="warning" >}}
리더-라이터에는 **기아**가 숨어 있다. 리더가 끊임없이 들어오면 라이터가 영원히 락을 못 얻는 **라이터 기아**가, 반대로 라이터를 우대하면 **리더 기아**가 생긴다. 어느 쪽을 우선할지(리더 우선·라이터 우선·공정)는 정책 선택이며, 읽기가 압도적으로 많고 쓰기가 드문 캐시 같은 곳에서 리더-라이터 락의 이점이 가장 크다.
{{< /callout >}}

## 7. 정리

```
┌──────────────────────────────────────────────────────────────┐
│         교착 상태와 기아 상태 한눈에 보기                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  활성(liveness) 실패 3형제                                   │
│   ├ 교착   : 서로 상대 자원 기다리며 멈춤 (순환 대기)        │
│   ├ 라이브락: 양보·재시도만 반복, 진행 0                     │
│   └ 기아   : 특정 작업이 자원을 못 받아 굶음                 │
│                                                              │
│  교착 4조건 (하나만 깨도 예방)                               │
│   상호배제·점유와대기·비선점·순환대기                        │
│   → 실무: 락 획득 순서 통일이 정석                           │
│                                                              │
│  고전 문제와 해법                                            │
│   ├ 프로듀서-컨슈머 : 세마포어 2개(빈칸·찬칸)+뮤텍스        │
│   └ 리더-라이터     : 읽기 공유·쓰기 독점 락                 │
└──────────────────────────────────────────────────────────────┘
```

### 한 줄 정리

| 개념 | 한 줄 |
|:-----|:------|
| **교착 상태** | "서로 상대의 자원을 기다리며 영원히 멈춤" |
| **교착 4조건** | "상호배제·점유와대기·비선점·순환대기 — 하나만 깨면 예방" |
| **락 순서 통일** | "모두 같은 순서로 락을 잡아 순환 대기를 없앤다" |
| **라이브락** | "바쁘게 양보·재시도하나 진행이 없음" |
| **기아 상태** | "욕심 많은 작업에 밀려 자원을 못 받아 굶음" |
| **공정성/에이징** | "오래 기다린 작업을 먼저·점점 우대해 기아 방지" |
| **프로듀서-컨슈머** | "유한 버퍼를 세마포어 2개로 조율" |
| **리더-라이터** | "읽기는 여럿 동시에, 쓰기는 홀로" |

{{< callout type="info" >}}
**핵심 정리:**
1. 동기화는 경쟁 조건을 막지만, 락을 부주의하게 쓰면 **활성 실패**(교착·라이브락·기아)를 부른다 — 값이 틀리는 게 아니라 일이 끝나지 않는 문제다
2. **교착 상태**는 서로가 상대의 자원을 기다리며 멈추는 것으로, **네 조건(상호배제·점유와대기·비선점·순환대기)** 이 모두 성립할 때만 생긴다 — 하나만 깨도 막을 수 있고, 실무에선 **락 획득 순서 통일**이 가장 실용적이다
3. **라이브락**은 서로 양보·재시도만 반복해 진행이 없는 상태로, **무작위 지연**으로 타이밍을 어긋나게 해 푼다
4. **기아 상태**는 특정 작업이 자원을 못 받아 굶는 것으로, **공정 락·에이징·우선순위 상속**으로 완화한다
5. **프로듀서-컨슈머**는 세마포어 2개로 빈칸·찬칸을 세어, **리더-라이터**는 읽기 공유·쓰기 독점 락으로 푸는 것이 정석이다
{{< /callout >}}
