작업을 잘게 쪼개 동시에 돌리기 시작하면(7장), 곧바로 새로운 문제가 따라온다 — 여러 작업이 **같은 공유 자원**을 동시에 건드리는 순간이다. 이 장에서는 그때 생기는 대표적 버그인 **경쟁 조건(race condition)**, 문제가 일어나는 코드 구역인 **임계 구역(critical section)**, 그리고 이를 막는 **동기화(synchronization)** 수단인 **원자적 연산 · 락 · 뮤텍스 · 세마포어**를 살펴본다. 마지막으로 동기화는 공짜가 아니라는 점, 즉 **비용**을 짚는다.

---

## 1. 공유 자원이 부르는 문제

### 1.1 동시 접근의 위험

동시성 프로그램은 거의 항상 **공유 자원**(메모리 변수, 파일, 커넥션, 카운터 등)을 함께 쓴다. 문제는 여러 작업의 실행이 **교차(interleaving)** 될 때 생긴다. 한 작업이 자원을 읽고 고쳐 쓰는 도중에 다른 작업이 끼어들면, 프로그램이 의도하지 않은 결과를 내거나 **지금은 멀쩡해 보여도 나중에 터지는 버그**를 심게 된다.

```
┌──────────────────────────────────────────────────────────────┐
│         교차 실행이 값을 망가뜨리는 과정                       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  공유 변수 counter = 0,  두 스레드가 +1씩 (기대값 2)         │
│                                                              │
│   스레드 A            스레드 B          counter             │
│   읽기(0) ─────────────────────────────  0                  │
│                      읽기(0) ──────────  0                  │
│   +1 → 1                                 0                  │
│   쓰기(1) ─────────────────────────────  1                  │
│                      +1 → 1                                 │
│                      쓰기(1) ──────────  1  ← 갱신 손실!     │
│                                                              │
│   결과 1  (기대 2)  → 한 번의 증가가 사라짐                  │
└──────────────────────────────────────────────────────────────┘
```

`counter++`는 한 줄처럼 보여도 실제로는 **읽기 → 더하기 → 쓰기** 세 단계로 쪼개진다. 이 세 단계 사이에 다른 스레드가 끼어들 수 있다는 것이 모든 문제의 출발점이다.

### 1.2 경쟁 조건

> **경쟁 조건(race condition)** 이란 두 개 이상의 작업이 공유 자원을 동시에 변경하려 할 때, **연산의 결과가 작업들의 접근 순서에 따라 달라지는** 상황을 말한다.

"경쟁"이라는 이름은 두 스레드가 자원을 두고 **경주(race)** 하는 모습에서 왔다. 누가 먼저 닿느냐에 따라 결과가 바뀌고, 그 순서는 스케줄러가 매번 다르게 정하므로 **재현이 어렵고 간헐적으로만 드러나는** 가장 악질적인 버그가 된다.

{{< callout type="warning" >}}
경쟁 조건은 **대부분의 실행에서 우연히 정답이 나온다**는 점이 특히 위험하다. 테스트에서는 통과하다가 운영 환경의 부하·타이밍에서만 어쩌다 틀린 값을 내므로, 디버거로 잡으려 해도 중단점이 타이밍을 바꿔 버려 증상이 사라진다(하이젠버그, Heisenbug). 그래서 "테스트로 없음을 증명"하기 어렵고, **설계로 막는 것**이 정석이다.
{{< /callout >}}

## 2. 임계 구역

### 2.1 정의

> **임계 구역(critical section)** 은 동시에 실행 중인 여러 작업이 같은 공유 자원에 접근을 시도할 수 있는 코드 구역이다.

경쟁 조건은 코드 아무 데서나 생기지 않는다. **공유 자원을 읽고 쓰는 바로 그 구간**에서만 생긴다. 이 구간을 임계 구역이라 부르고, 동기화의 목표는 단 하나다 — **임계 구역에는 한 번에 한 작업만 들어가게 하는 것(상호 배제)**.

```
┌──────────────────────────────────────────────────────────────┐
│         임계 구역과 상호 배제                                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   진입부          ┌─ 임계 구역 ─┐         퇴장부             │
│   (락 획득) ───▶  │ 공유 자원   │  ───▶  (락 해제)           │
│                   │ 읽기·변경   │                            │
│                   └─────────────┘                            │
│                    한 번에 한 작업만                          │
│                                                              │
│   다른 작업은 진입부에서 대기 ──▶ 앞 작업이 퇴장해야 진입    │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 좋은 임계 구역의 조건

임계 구역은 **꼭 필요한 만큼만 작게** 잡아야 한다. 너무 넓게 잡으면 사실상 직렬 실행이 되어 동시성의 이점이 사라지고(2장 암달의 법칙), 너무 좁게 잡아 공유 접근을 빠뜨리면 경쟁 조건이 새어 나온다.

| 임계 구역 설계 | 결과 |
|:--------------|:-----|
| 너무 넓음 | 직렬화 → 병렬성 손실, 처리율 하락 |
| 너무 좁음 | 보호 누락 → 경쟁 조건 잔존 |
| 적정(공유 접근만 감쌈) | 안전하면서 병렬성 유지 |

## 3. 동기화 수단 ① 원자적 연산

### 3.1 원자성이란

임계 구역의 의도치 않은 동작을 막는 **가장 단순한 방법**은 연산 자체를 쪼갤 수 없게 만드는 것이다.

> **원자적(atomic)** 연산이란 그 **중간 단계를 다른 스레드가 결코 볼 수 없는** 연산이다. "전부 일어났거나, 전혀 일어나지 않았거나" 둘 중 하나만 관측된다.

`counter++`가 위험했던 이유는 읽기·더하기·쓰기 사이가 들여다보였기 때문이다. 이 셋을 하드웨어가 보장하는 **하나의 쪼갤 수 없는 연산(CAS 등)** 으로 묶으면 경쟁 조건이 사라진다.

```java
// 원자적 연산 — 하드웨어 CAS로 읽기·더하기·쓰기를 한 덩어리로
AtomicLong counter = new AtomicLong(0);

// 여러 스레드가 동시에 호출해도 갱신 손실이 없다
counter.incrementAndGet();   // 내부적으로 compareAndSet 재시도 루프
```

### 3.2 한계

원자적 연산은 빠르고 락이 없어 매력적이지만, **하드웨어와 런타임이 지원하는 단순 연산**(증가, 교체, CAS)에만 쓸 수 있다. 여러 변수를 한꺼번에 일관되게 바꾸는 복합 연산은 원자적 연산만으로 표현할 수 없어, 결국 락 같은 다른 수단이 필요하다.

{{< callout type="info" >}}
"원자적"과 "스레드 안전"은 같지 않다. `if (map.get(k) == null) map.put(k, v)` 처럼 **개별 연산이 각각 원자적이어도, 그 둘을 잇는 검사-후-실행(check-then-act)은 원자적이지 않다.** 이 틈에 다른 스레드가 끼어든다. 이럴 때는 `putIfAbsent` 같은 **복합 동작 자체를 원자화한 API**를 쓰거나 락으로 감싸야 한다.
{{< /callout >}}

## 4. 동기화 수단 ② 락

> **락(lock)** 은 공유 자원에 걸어 두는 추상적인 자물쇠다. 락을 **획득(acquire)** 한 스레드만 자원에 접근할 수 있고, 락을 얻지 못한 스레드는 앞선 스레드가 락을 **해제(release)** 할 때까지 기다린다.

락은 임계 구역의 **진입부에서 획득하고 퇴장부에서 해제**하는 식으로 상호 배제를 구현하는 일반적 메커니즘이다. 뮤텍스·세마포어는 이 락 개념을 구체화한 특수한 형태로 볼 수 있다.

```java
Lock lock = new ReentrantLock();

lock.lock();              // 진입부 — 락 획득(없으면 대기)
try {
    balance += amount;    // ── 임계 구역 ──
} finally {
    lock.unlock();        // 퇴장부 — 반드시 해제
}
```

{{< callout type="warning" >}}
락 해제는 **반드시 `finally`에서** 한다. 임계 구역 안에서 예외가 터졌을 때 락을 풀지 않으면, 그 락을 기다리던 모든 스레드가 영원히 멈춰 선다(데드락에 준하는 영구 대기). 또 여러 락을 **서로 다른 순서로** 잡으면 데드락이 생긴다 — 락 획득 순서를 전역에서 하나로 통일하는 것이 기본 예방책이다.
{{< /callout >}}

## 5. 동기화 수단 ③ 뮤텍스

### 5.1 상호 배제의 자물쇠

어떤 작업이 공유 자원을 **읽고 변경하는 동안 다른 작업이 끼어들지 못하게** 해야 하는 연산이 있다. 이런 **상호 배제(mutual exclusion)** 가 필요할 때 쓰는 락이 **뮤텍스(mutex)** 다 — 이름 자체가 *MUTual EXclusion*의 줄임말이다.

뮤텍스의 핵심 규칙은 **소유권(ownership)** 이다. **락을 건 작업만이 그 락을 풀 수 있다.** 자원을 "내가 점유했다"는 소유 개념이 있어, 화장실 칸 하나에 열쇠 하나가 달린 구조에 비유된다.

```
┌──────────────────────────────────────────────────────────────┐
│         뮤텍스 — 한 명만, 잠근 사람이 연다                     │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   [자원] ── 열쇠 1개 ──                                       │
│                                                              │
│   스레드 A : 잠금 ▶ 사용 ▶ 해제   (A만 해제 가능)            │
│   스레드 B : ─── 대기 ───────────▶ 잠금 ▶ 사용 ▶ 해제       │
│                                                              │
│   동시 접근 허용 수 = 1                                       │
└──────────────────────────────────────────────────────────────┘
```

## 6. 동기화 수단 ④ 세마포어

### 6.1 여럿을 허용하는 신호기

> **세마포어(semaphore)** 도 공유 자원 접근을 제어하지만, **여러 작업이 동시에 접근하도록 허용**한다는 점에서 뮤텍스와 다르다.

세마포어는 **허가증(permit) 개수**를 들고 있는 카운터다. 접근하려는 작업은 허가증을 하나 얻고(`acquire`), 다 쓰면 반납한다(`release`). 허가증이 N개면 최대 N개 작업이 동시에 자원을 쓸 수 있다. 허가증이 떨어지면 다음 작업은 누군가 반납할 때까지 기다린다.

뮤텍스와의 또 다른 결정적 차이는 **소유권이 없다**는 점이다. 그래서 **락을 건 작업이 아닌 다른 작업도 허가증을 반납할 수 있다.** 신호기(철도의 통행 신호)처럼, "지금 몇 대까지 들어와도 되는가"라는 신호만 관리할 뿐 누가 들어왔는지는 따지지 않는다.

```
┌──────────────────────────────────────────────────────────────┐
│         세마포어 — 허가증 N개, 동시 N명 허용                   │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   [자원 풀] ── 허가증 3개 ──                                  │
│                                                              │
│   A acquire ▶ 사용   (남은 2)                                │
│   B acquire ▶ 사용   (남은 1)                                │
│   C acquire ▶ 사용   (남은 0)                                │
│   D acquire ▶ ── 대기 ──  (누군가 release 해야 진입)         │
│                                                              │
│   소유권 없음 → 획득한 작업이 아니어도 release 가능           │
└──────────────────────────────────────────────────────────────┘
```

```java
// 세마포어 — DB 커넥션을 동시 3개까지만
Semaphore sem = new Semaphore(3);

sem.acquire();            // 허가증 획득(없으면 대기)
try {
    useConnection();      // 최대 3개 작업이 동시에 여기
} finally {
    sem.release();        // 허가증 반납
}
```

### 6.2 뮤텍스 vs 세마포어

| 구분 | 뮤텍스(mutex) | 세마포어(semaphore) |
|:-----|:-------------|:--------------------|
| 동시 접근 허용 수 | 1 | N (허가증 개수) |
| 소유권 | 있음 (잠근 작업만 해제) | 없음 (누구나 반납 가능) |
| 주 용도 | 상호 배제 | 자원 풀·동시성 제한 |
| 비유 | 열쇠 1개 달린 화장실 | N자리 주차장 신호기 |

{{< callout type="info" >}}
허가증 1개짜리 세마포어(`Semaphore(1)`)는 뮤텍스처럼 보이지만 **소유권이 없어** 완전히 같지는 않다. 한 스레드가 `acquire`하고 *다른* 스레드가 `release`해도 막지 못해, 상호 배제 의도를 어기는 실수를 잡아 주지 못한다. 순수한 상호 배제에는 뮤텍스(또는 `ReentrantLock`)를, 동시 실행 *수를 제한*하는 데는 세마포어를 쓰는 것이 의도를 분명히 드러낸다.
{{< /callout >}}

## 7. 동기화는 공짜가 아니다

동기화는 안전을 사 주지만 **비용**을 치르게 한다. 락을 잡고 푸는 연산, 대기하는 스레드의 블로킹과 컨텍스트 스위칭(4·6장), 임계 구역에서 강제되는 직렬 실행이 모두 성능을 갉아먹는다.

```
┌──────────────────────────────────────────────────────────────┐
│         동기화의 비용                                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   ① 락 획득·해제 연산 자체의 오버헤드                        │
│   ② 대기 스레드의 블로킹 → 컨텍스트 스위칭 비용              │
│   ③ 임계 구역 = 직렬 실행 → 병렬성 손실(암달의 법칙)         │
│                                                              │
│   원칙: 꼭 필요한 곳에만, 임계 구역은 최대한 좁게            │
└──────────────────────────────────────────────────────────────┘
```

그래서 동기화는 **정말 필요한 경우에만** 사용한다. 더 나아가, 애초에 공유 상태를 만들지 않는 설계 — **불변 객체**, 작업마다 **데이터를 분리**해 갖기, **메시지 전달**(5장 IPC)로 자원을 직접 공유하지 않기 — 가 가장 싸고 안전한 해법인 경우가 많다.

| 경쟁 조건 해소 수단 | 핵심 |
|:-------------------|:-----|
| 원자적 연산 | 단순 연산을 쪼갤 수 없게 (락 없음, 가장 쌈) |
| 락 / 뮤텍스 | 임계 구역에 한 작업만 (상호 배제) |
| 세마포어 | 동시 접근 수를 N으로 제한 |
| 메시지 전달 · IPC | 자원을 공유하지 않고 주고받음 |
| 불변 · 데이터 분리 | 공유 상태 자체를 없앰 (동기화 불필요) |

## 8. 정리

```
┌──────────────────────────────────────────────────────────────┐
│         경쟁 조건과 동기화 한눈에 보기                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  공유 자원 + 교차 실행 → 경쟁 조건                           │
│   └ 결과가 접근 순서에 좌우, 재현 어려움                      │
│                                                              │
│  문제 구역 = 임계 구역                                       │
│   └ 목표: 한 번에 한 작업만 (상호 배제), 작게 잡기           │
│                                                              │
│  동기화 수단                                                 │
│   ├ 원자적 연산 : 쪼갤 수 없는 단순 연산 (락 없음)          │
│   ├ 락          : 획득·해제로 상호 배제                      │
│   ├ 뮤텍스      : 동시 1, 소유권 있음                        │
│   └ 세마포어    : 동시 N, 소유권 없음                        │
│                                                              │
│  동기화는 비용 → 꼭 필요한 곳에만, 임계 구역은 좁게          │
└──────────────────────────────────────────────────────────────┘
```

### 한 줄 정리

| 개념 | 한 줄 |
|:-----|:------|
| **경쟁 조건** | "결과가 접근 순서에 따라 달라지는 상황" |
| **임계 구역** | "공유 자원을 건드리는 코드 구역 — 한 번에 한 작업만" |
| **원자적 연산** | "중간이 안 보이는 쪼갤 수 없는 연산" |
| **락** | "획득한 작업만 자원에 접근" |
| **뮤텍스** | "동시 1명, 잠근 사람이 연다(소유권)" |
| **세마포어** | "동시 N명, 누구나 반납 가능(소유권 없음)" |
| **동기화 비용** | "안전엔 값이 있다 — 꼭 필요한 곳에만" |

{{< callout type="info" >}}
**핵심 정리:**
1. **경쟁 조건**은 여러 작업이 공유 자원을 동시에 변경할 때 결과가 접근 순서에 좌우되는 버그로, 간헐적으로만 드러나 재현·디버깅이 어렵다 — 설계로 막는 것이 정석이다
2. 문제는 공유 자원을 건드리는 **임계 구역**에서만 생기며, 동기화의 목표는 임계 구역에 **한 번에 한 작업만** 들이는 **상호 배제**다 — 구역은 꼭 필요한 만큼만 작게 잡는다
3. **원자적 연산**은 중간 단계가 보이지 않는 쪼갤 수 없는 연산으로 가장 싸지만, 하드웨어가 지원하는 단순 연산에만 쓸 수 있다
4. **락**은 획득·해제로 상호 배제를 구현하는 일반 수단이고, **뮤텍스**는 동시 1·소유권 있음, **세마포어**는 동시 N·소유권 없음으로 용도가 갈린다
5. **동기화는 비용**(락 연산·블로킹·직렬화)을 치르므로 꼭 필요한 곳에만 쓰고, 공유 상태 자체를 없애는 불변·데이터 분리·메시지 전달이 더 싸고 안전할 때가 많다
