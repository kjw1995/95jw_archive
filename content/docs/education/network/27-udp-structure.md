## 1. UDP의 개념

UDP(User Datagram Protocol)는 전송 계층의 비연결형 프로토콜이다. 신뢰성·순서 보장 같은 기능을 과감히 포기한 대신 지연과 오버헤드를 최소화한다.

핵심 특성은 다음과 같다.

- **비연결형**: 연결 수립·해제 절차 없음
- **비신뢰성**: 손실·중복·순서 뒤바뀜 모두 가능
- **저오버헤드**: 헤더 8바이트 고정
- **빠른 전송**: ACK 대기 없음
- **브로드캐스트·멀티캐스트 지원**

이름대로 "사용자 데이터그램"을 전송한다. 각 데이터그램은 독립적으로 처리되며 서로 아무런 관계가 없다.

## 2. TCP와 비교

| 항목 | TCP | UDP |
|:---|:---|:---|
| 연결 | 3-way handshake | 없음 |
| 신뢰성 | ACK, 재전송 | 없음 |
| 순서 | 보장 | 미보장 |
| 헤더 | 20~60 바이트 | 8 바이트 |
| 속도 | 느림 | 빠름 |
| 흐름/혼잡 제어 | 있음 | 없음 |
| 브로드캐스트 | 불가 | 가능 |
| 통신 방식 | 1:1 | 1:1, 1:N, N:N |
| 데이터 단위 | 세그먼트 | 데이터그램 |

비유하면 TCP는 등기우편, UDP는 엽서다. 엽서는 수취 확인을 하지 않으므로 잃어버려도 재발송이 없지만 그만큼 빠르고 비용이 적다.

## 3. UDP 헤더 구조

헤더는 단 8바이트, 4개 필드로 구성된다.

```text
 0        16        31
 ├─────────┼─────────┤
 │ src port│ dst port│
 ├─────────┼─────────┤
 │  length │checksum │
 ├─────────┴─────────┤
 │       data        │
 └───────────────────┘
```

| 필드 | 크기 | 설명 |
|:---|:---|:---|
| Source Port | 16b | 송신 애플리케이션 포트 (응답 필요 없으면 0 가능) |
| Destination Port | 16b | 수신 애플리케이션 포트 (필수) |
| Length | 16b | UDP 헤더 + 데이터의 전체 바이트 수 (최소 8) |
| Checksum | 16b | 오류 검출, IPv4에서는 선택이지만 IPv6에서는 필수 |

### 의사 헤더

체크섬 계산에는 IP 헤더의 일부(출발지·목적지 IP, 프로토콜=17, UDP 길이)를 포함한다. 이 의사 헤더 덕에 IP 주소가 잘못 라우팅된 경우에도 검출할 수 있다.

### TCP와 헤더 비교

TCP에 있지만 UDP에는 없는 필드들이다.

- Sequence / Acknowledgment Number → 순서·신뢰성 없음
- Flags (SYN, ACK, FIN) → 연결 개념 없음
- Window Size → 흐름 제어 없음
- Options → 확장 기능 없음

## 4. UDP 데이터그램

UDP 헤더가 붙은 데이터 단위를 데이터그램이라고 한다.

```text
애플리케이션 데이터
   ↓ UDP 헤더 추가
[UDP 헤더 | 데이터]            ← UDP 데이터그램
   ↓ IP 헤더 추가
[IP 헤더 | UDP 헤더 | 데이터]  ← IP 패킷
```

각 데이터그램은 자기 완결적이며, 서로 순서 관계가 없다. TCP 세그먼트가 "하나의 연결 내부 구성 요소"라면 UDP 데이터그램은 "각자 독립된 소포"에 가깝다.

## 5. UDP 통신 과정

### TCP와의 단계 수 비교

```text
TCP: SYN → SYN+ACK → ACK → DATA → ACK → FIN → FIN+ACK
     (연결/데이터/종료에 최소 7단계)

UDP: DATA
     (데이터 한 번에 끝)
```

### DNS 쿼리 예시

```text
Client 192.168.1.10:52341    DNS 8.8.8.8:53
    │                              │
    │── "google.com A?"  ────────→│
    │                              │
    │←── "142.250.185.78" ────────│
    │                              │
```

연결 수립 단계가 없으므로 작은 요청/응답은 거의 왕복 시간만 소요된다. 실패하면 애플리케이션 레벨에서 재시도한다.

## 6. 브로드캐스트와 멀티캐스트

UDP는 1:N 통신을 지원한다. TCP로는 불가능한 통신 방식이다.

### 유니캐스트 vs 브로드캐스트 vs 멀티캐스트

| 구분 | 대상 | 프로토콜 | 특징 |
|:---|:---|:---|:---|
| Unicast | 1:1 | TCP, UDP | 일반 통신 |
| Broadcast | 1:All (같은 LAN) | UDP | 라우터 통과 불가 |
| Multicast | 1:Group | UDP | 그룹 가입자만 수신, 라우터 통과 가능 |

### 브로드캐스트 예시

DHCP는 IP가 없는 상태에서 서버를 찾아야 하므로 브로드캐스트가 필수다.

```text
New PC ─ "DHCP 서버 있나요?" (255.255.255.255:67) ─→ 모든 장비
       ←── "192.168.1.100 쓰세요" ───── DHCP 서버
```

### 멀티캐스트 예시

IPTV는 같은 채널을 보는 시청자만 멀티캐스트 그룹에 가입시킨다. 송신자는 하나의 스트림만 보내고 네트워크가 가입자에게 복제 전달한다.

```text
Streaming Server ─ 239.1.1.1:5000 ─→ Router
                                      ├─→ 가입자 A
                                      ├─→ 가입자 B
                                      └─× 비가입자 (받지 않음)
```

## 7. UDP를 사용하는 주요 프로토콜

| 프로토콜 | 포트 | 사용 이유 |
|:---|:---|:---|
| DNS | 53 | 작은 요청/응답, 빠른 해석 |
| DHCP | 67/68 | 브로드캐스트 필요 |
| TFTP | 69 | 단순 파일 전송, 임베디드 |
| SNMP | 161/162 | 네트워크 모니터링 |
| NTP | 123 | 시간 동기화, 오버헤드 최소화 |
| RTP | 가변 | 실시간 오디오/비디오 |
| VoIP | 가변 | 음성 통화, 실시간성 우선 |

{{< callout type="info" >}}
재전송한 음성/영상 프레임은 이미 지나간 시점이라 틀어도 의미가 없다. 이런 실시간 미디어에는 TCP보다 UDP가 맞다.
{{< /callout >}}

## 8. UDP가 적합한 상황과 부적합한 상황

### UDP가 적합한 경우

- **실시간성이 중요**: 스트리밍, 게임, VoIP
- **짧은 요청/응답**: DNS, DHCP, NTP
- **브로드캐스트/멀티캐스트**: mDNS, IPTV, 서비스 탐색
- **애플리케이션이 자체 재전송 로직을 가진 경우**: 커스텀 프로토콜

### UDP가 부적합한 경우

- **손실이 허용되지 않는 경우**: 파일 다운로드, 이메일, 금융 거래
- **순서가 중요한 경우**: HTTP 응답, 데이터베이스 쿼리
- **대용량 전송**: 업로드, 백업

## 9. 장단점 정리

### 장점

- 낮은 지연 (연결 설정 없음)
- 낮은 오버헤드 (헤더 8바이트)
- 브로드캐스트·멀티캐스트 지원
- 구현이 단순

### 단점

- 패킷 손실 가능, 재전송 없음
- 순서 보장 없음
- 흐름·혼잡 제어 없음 (네트워크 악화 시에도 계속 송신)
- 연결 상태 없음 (상대 생존 여부 모름)

{{< callout type="warning" >}}
UDP는 혼잡 제어가 없기 때문에 잘못 설계하면 네트워크를 포화시킬 수 있다. 실시간 스트리밍 서비스라면 애플리케이션 레벨에서 비트레이트 조절 같은 대책을 함께 둔다.
{{< /callout >}}

## 10. 소켓 프로그래밍 예시 (Python)

### 서버

```python
import socket

server = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
server.bind(('0.0.0.0', 9999))

while True:
    data, addr = server.recvfrom(1024)
    print(f"from {addr}: {data.decode()}")
    server.sendto(f"echo: {data.decode()}".encode(), addr)
```

### 클라이언트

```python
import socket

client = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
client.sendto("hello UDP".encode(), ('localhost', 9999))

data, server = client.recvfrom(1024)
print(data.decode())
client.close()
```

TCP와 비교하면 `listen()`, `accept()`, `connect()` 같은 연결 단계가 없다.

## 11. UDP 확인 명령어

```bash
# Windows
netstat -an -p UDP

# Linux
ss -u -a
# UNCONN  0  0  0.0.0.0:53  0.0.0.0:*
netstat -u -a
```

`UNCONN` 상태는 비연결형을 뜻한다. UDP는 연결 상태 자체가 없어 TCP처럼 `ESTABLISHED` 같은 상태가 표시되지 않는다.

### Wireshark 필터

```text
udp
udp.port == 53                                    # DNS
udp.port == 67 || udp.port == 68                  # DHCP
ip.dst == 255.255.255.255                         # 브로드캐스트
ip.dst >= 224.0.0.0 && ip.dst <= 239.255.255.255  # 멀티캐스트
```

## 핵심 정리

| 개념 | 설명 |
|:---|:---|
| UDP | 비연결형, 빠른 전송 계층 프로토콜 |
| 데이터그램 | 독립적인 UDP 데이터 단위 |
| 헤더 | 8바이트 고정 (src/dst port, length, checksum) |
| 비신뢰성 | 손실·순서 뒤바뀜·중복 가능 |
| 브로드캐스트 | 같은 LAN 전체 전송 |
| 멀티캐스트 | 그룹 가입자만 수신 |
| 사용처 | DNS, DHCP, NTP, 스트리밍, 게임, VoIP |

{{< callout type="info" >}}
**용어 정리**
- **UDP (User Datagram Protocol)**: 비연결형, 비신뢰성 전송 계층 프로토콜
- **데이터그램 (Datagram)**: UDP 헤더가 붙은 독립 데이터 단위
- **비연결형 (Connectionless)**: 연결 수립 없이 전송하는 방식
- **유니캐스트 / 브로드캐스트 / 멀티캐스트**: 1:1 / 1:All / 1:Group 전송
- **체크섬 (Checksum)**: 오류 검출 값
- **의사 헤더 (Pseudo Header)**: IP 헤더 정보를 포함해 체크섬을 계산하는 가상 헤더
{{< /callout >}}
