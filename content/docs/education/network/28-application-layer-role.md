## 1. 응용 계층이란?

### 정의와 역할

응용 계층(Application Layer)은 OSI 7계층 모델의 최상위 계층으로, 사용자 애플리케이션에 네트워크 서비스를 제공한다. 웹 브라우저, 이메일 클라이언트, FTP 클라이언트 같은 프로그램은 이 계층에서 동작하는 프로토콜을 통해 원격 서버와 통신한다.

- 사용자와 가장 가까운 계층
- 애플리케이션 간 통신 규칙 정의
- 데이터 형식과 전송 방법 규정
- 하위 계층의 복잡성 추상화

### OSI 7계층에서의 위치

```
┌────────────────────────────┐
│ 7. 응용 (Application)      │ ← 이 장
│    HTTP, DNS, SMTP ...     │
├────────────────────────────┤
│ 6. 표현 (Presentation)     │
│    SSL/TLS, 인코딩         │
├────────────────────────────┤
│ 5. 세션 (Session)          │
├────────────────────────────┤
│ 4. 전송 (Transport)        │
│    TCP, UDP                │
├────────────────────────────┤
│ 3. 네트워크 (Network)      │
│    IP                      │
├────────────────────────────┤
│ 2. 데이터링크 (Data Link)  │
│    Ethernet, MAC           │
├────────────────────────────┤
│ 1. 물리 (Physical)         │
└────────────────────────────┘
```

### 계층별 데이터 단위

| 계층 | 데이터 단위 |
|:---|:---|
| 응용 | 데이터, 메시지 |
| 전송 | 세그먼트, 데이터그램 |
| 네트워크 | 패킷 |
| 데이터링크 | 프레임 |
| 물리 | 비트 |

## 2. 클라이언트-서버 모델

### 역할 구분

응용 계층 통신은 대부분 **요청-응답(request-response)** 패턴의 클라이언트-서버 모델로 동작한다.

| 구분 | 역할 | 예시 |
|:---|:---|:---|
| 클라이언트 | 서비스 요청, 능동적 연결 | 웹 브라우저, 메일 클라이언트 |
| 서버 | 서비스 제공, 수동적 대기 | Apache, Postfix, BIND |

### 요청-응답 흐름

```
[클라이언트]         [서버]
    │  1. Request     │
    │ ───────────────▶│
    │                 │ 처리
    │  2. Response    │
    │ ◀─────────────── │
    │                 │
```

{{< callout type="info" >}}
클라이언트가 먼저 연결을 시도하고, 서버는 특정 포트에서 연결을 대기(LISTEN)한다. HTTP는 80, HTTPS는 443, SSH는 22 같은 Well-Known Port가 이런 "만나는 약속"을 만들어준다.
{{< /callout >}}

## 3. 주요 응용 계층 프로토콜

### HTTP / HTTPS

웹 페이지 전송용 프로토콜이다. TCP 위에서 동작하며 상태를 유지하지 않는 요청-응답 구조를 사용한다.

```
GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Accept: text/html
```

- HTTP: 포트 80, 평문
- HTTPS: 포트 443, TLS 암호화

### DNS

도메인 이름을 IP 주소로 변환한다. 짧은 질의와 응답이 많아 주로 UDP(포트 53)를 사용하며, 영역 전송이나 큰 응답에는 TCP를 쓴다.

### FTP

파일 전송 프로토콜이다. 제어 연결(21)과 데이터 연결(20)을 분리하여 명령과 파일 전송을 별도로 수행한다. 평문 전송이므로 실제 환경에서는 SFTP나 FTPS를 권장한다.

### SMTP / POP3 / IMAP

메일 시스템에서 송신과 수신이 서로 다른 프로토콜로 처리된다. 자세한 내용은 [31. 메일 서버의 구조](../31-mail-server-structure)에서 다룬다.

| 프로토콜 | 역할 | 포트 |
|:---|:---|:---|
| SMTP | 메일 송신, 서버 간 릴레이 | 25, 587 |
| POP3 | 메일 수신 (다운로드) | 110, 995 |
| IMAP | 메일 수신 (서버 보관, 동기화) | 143, 993 |

### 프로토콜과 포트 매핑

| 프로토콜 | 용도 | 포트 | 전송 계층 |
|:---|:---|:---|:---|
| HTTP | 웹 (평문) | 80 | TCP |
| HTTPS | 웹 (암호화) | 443 | TCP |
| DNS | 이름 해석 | 53 | UDP (주) / TCP |
| FTP | 파일 전송 | 20, 21 | TCP |
| SMTP | 메일 송신 | 25, 587 | TCP |
| POP3 | 메일 수신 | 110 | TCP |
| IMAP | 메일 수신 | 143 | TCP |
| SSH | 원격 셸 | 22 | TCP |
| DHCP | IP 할당 | 67, 68 | UDP |
| NTP | 시각 동기화 | 123 | UDP |

### 전송 계층 선택 기준

| 특성 | TCP 선택 | UDP 선택 |
|:---|:---|:---|
| 신뢰성 | 손실 불가 | 손실 허용 |
| 대표 프로토콜 | HTTP, FTP, SMTP | DNS, DHCP, NTP |
| 이유 | 순서·무결성 | 빠른 응답, 브로드캐스트 |

## 4. 캡슐화와 역캡슐화

송신 측에서는 응용 데이터가 하위 계층으로 내려가며 각 계층의 헤더가 붙는다. 수신 측에서는 반대로 헤더가 하나씩 제거된다.

```
[송신]                 [수신]
응용 데이터             응용 데이터
    │                      ▲
    ▼ +TCP                 │ -TCP
 세그먼트               세그먼트
    │                      ▲
    ▼ +IP                  │ -IP
  패킷                    패킷
    │                      ▲
    ▼ +이더넷              │ -이더넷
 프레임                  프레임
    │                      ▲
    ▼ 전기 신호            │
    └──────────────────────┘
```

## 5. 실무 도구

### HTTP 확인

```bash
curl -v http://www.example.com
curl -I http://www.example.com   # 헤더만
```

### DNS 조회

```bash
nslookup www.google.com
dig www.google.com +short
host www.google.com
```

### 메일 프로토콜 수동 테스트

```bash
telnet smtp.example.com 25
openssl s_client -connect pop.example.com:995
```

## 핵심 정리

| 항목 | 내용 |
|:---|:---|
| 계층 | OSI 7계층 중 최상위 |
| 통신 모델 | 클라이언트-서버, 요청-응답 |
| 대표 프로토콜 | HTTP, DNS, SMTP, POP3, IMAP, FTP |
| 전송 계층 | 신뢰성이 필요하면 TCP, 속도가 우선이면 UDP |

{{< callout type="info" >}}
**용어 정리**
- **응용 계층**: 사용자 애플리케이션에 네트워크 서비스를 제공하는 최상위 계층
- **Well-Known Port**: 0~1023의 표준 포트 (HTTP 80, HTTPS 443 등)
- **요청-응답 패턴**: 클라이언트가 요청을 보내고 서버가 응답하는 통신 구조
{{< /callout >}}
