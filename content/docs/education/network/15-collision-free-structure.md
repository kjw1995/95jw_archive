## 1. 전이중과 반이중 통신

### 두 통신 방식

| 항목 | 전이중 (Full) | 반이중 (Half) |
|:---|:---|:---|
| 송·수신 | 동시에 수행 | 번갈아 수행 |
| 회선 | 송·수신 분리 | 공유 |
| 충돌 | 발생하지 않음 | 발생 가능 |
| CSMA/CD | 불필요 | 필요 |
| 대표 장비 | 스위치 | 허브 |
| 비유 | 전화 | 워키토키 |

### 원리

전이중은 송신 회선과 수신 회선이 물리적으로 분리되어 있어 동시에 양방향으로 신호를 보낼 수 있다. 반이중은 하나의 회선을 공유하므로 동시에 보내면 충돌이 일어난다.

```
전이중
[A] ══ 송신 ══→ [B]
[A] ←══ 수신 ══ [B]

반이중
[A] ←── 공유 회선 ──→ [B]
```

## 2. UTP 케이블의 송·수신 구조

### 100 Mbps 핀 배정

10/100 Mbps 이더넷은 UTP 8선 중 2쌍만 사용한다.

| 핀 | 용도 |
|:---|:---|
| 1, 2 | 송신 (TX+, TX-) |
| 3, 6 | 수신 (RX+, RX-) |
| 4, 5, 7, 8 | 미사용 |

송신 쌍과 수신 쌍이 분리되어 있으므로 PC-스위치 간에는 자연스럽게 전이중 통신이 가능하다.

### 1000 Mbps 이상

1000BASE-T는 4쌍 모두 사용하며, 각 쌍에서 양방향 신호가 함께 흐른다. 수신측은 에코 캔슬레이션 기술로 자기 송신 신호를 상쇄해 상대 신호만 복원한다.

## 3. 허브와 스위치의 통신 방식 차이

### 허브: 반이중

허브 내부의 공유 버스에는 모든 포트가 물려 있다. 둘 이상이 동시에 전송하면 버스에서 신호가 충돌한다.

```
  ┌────── 공유 버스 ──────┐
  │                       │
  P1  P2  P3  P4
```

### 스위치: 전이중

스위치는 포트마다 독립된 송·수신 버퍼와 스위칭 패브릭을 거쳐 프레임을 전달하므로 여러 포트가 동시에 통신해도 서로 간섭하지 않는다.

```
[A]──→ [SWITCH] ──→[C]
[B]──→         ──→[D]
  (동시 전송)
```

## 4. 콜리전 도메인 분리

### 허브: 모두 하나

```
     [HUB]
   │  │  │  │
  P1 P2 P3 P4
   └──┘  └──┘
 전체가 하나의 콜리전 도메인
```

### 스위치: 포트별로 분리

```
  [P1 CD] [P2 CD] [P3 CD]
      └──[SWITCH]──┘
```

각 포트가 독립된 콜리전 도메인이므로 실제 충돌이 발생하지 않는다. 전이중 모드에서는 CSMA/CD도 비활성화된다.

### 효율 비교 (100 Mbps, PC 4대)

| 환경 | 대역폭 배분 | 실효 성능 |
|:---|:---|:---|
| 허브 | 공유 100 M | 약 15 Mbps/대 (충돌 포함) |
| 스위치 | 포트별 100 M | 95 Mbps+ /대 (전이중 시 약 200 M) |

## 5. 브로드캐스트 도메인

스위치는 콜리전 도메인은 나누지만 브로드캐스트는 모든 포트로 전달한다. 브로드캐스트 영역을 나누려면 **라우터** 또는 **VLAN**이 필요하다.

| 장비 | 콜리전 도메인 | 브로드캐스트 도메인 |
|:---|:---|:---|
| 허브 | 하나 | 하나 |
| 스위치 | 포트별 | 하나 |
| 라우터 | 포트별 | 포트별 |

{{< callout type="warning" >}}
브로드캐스트 도메인이 커지면 ARP, DHCP 같은 브로드캐스트 트래픽이 늘어나 호스트마다 처리 부담이 증가한다. 중·대규모 네트워크에서는 VLAN을 이용해 논리적으로 도메인을 나눈다.
{{< /callout >}}

## 6. ARP: IP → MAC 변환

### 필요한 이유

이더넷 프레임을 만들려면 목적지 MAC 주소가 필요하다. 상위 계층은 IP 주소만 알고 있으므로 IP에 대응하는 MAC을 찾는 절차가 필요하다. 이것이 ARP이다.

### 동작 과정

```
① 요청: Broadcast
   "192.168.1.3 의 MAC?"
   목적지 MAC = FF:FF:FF:FF:FF:FF

② 응답: Unicast
   "192.168.1.3 = CC:CC:CC:CC:CC:03"

③ 캐시 저장
   ARP 테이블에 등록
```

| 계층 | 정보 |
|:---|:---|
| EtherType | 0x0806 |
| 요청 타입 | Request(1) / Reply(2) |

### ARP 캐시 확인

```bash
# Windows
arp -a

# Linux
ip neigh show
```

캐시 타임아웃은 Windows 기본 2분(사용 중이면 10분), Linux 60초, Cisco 4시간 등 OS·장비별로 다르다.

{{< callout type="info" >}}
ARP는 같은 브로드캐스트 도메인 내에서만 동작한다. 목적지 IP가 다른 네트워크에 있으면 자신의 기본 게이트웨이(라우터)의 MAC을 조회하고, 그 이후 구간은 라우터가 다시 ARP로 다음 홉을 찾는다.
{{< /callout >}}

## 7. 현대 네트워크의 구성 원칙

| 원칙 | 이유 |
|:---|:---|
| 허브 사용하지 않음 | 충돌·성능·보안 문제 |
| 스위치가 기본 장비 | 전이중, 독립 대역폭 |
| 라우터/VLAN으로 도메인 분리 | 브로드캐스트 최소화 |

---

## 핵심 정리

| 개념 | 설명 |
|:---|:---|
| 전이중 통신 | 송·수신 동시 수행, 충돌 없음 |
| 반이중 통신 | 송·수신 교대, 허브 환경 |
| 콜리전 도메인 분리 | 스위치 포트별로 분리 |
| 브로드캐스트 도메인 | 라우터/VLAN으로 분리 |
| ARP | IP 주소로 MAC을 찾는 프로토콜 |

{{< callout type="info" >}}
**용어 정리**
- **전이중 / 반이중**: 송·수신 동시 / 교대
- **콜리전 도메인**: 충돌 영향이 미치는 범위
- **브로드캐스트 도메인**: 브로드캐스트 프레임이 도달하는 범위
- **ARP**: IP 주소 → MAC 주소 변환 프로토콜
- **ARP 캐시**: IP-MAC 매핑 임시 저장소
{{< /callout >}}
