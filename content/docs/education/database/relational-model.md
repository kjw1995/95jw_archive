## 1. 관계 데이터 모델의 개념

관계 데이터 모델은 하나의 개체에 관한 데이터를 릴레이션(relation) 하나에 담아 데이터베이스에 저장하는 논리적 데이터 모델이다. 2차원 테이블 형태로 데이터를 표현하고, 테이블 간 관계는 외래 키로 연결한다.

### 릴레이션 예제

```text
고객 (릴레이션)
┌─────┬──────┬──────┬──────────────┐
│고객 │이름  │등급  │ 연락처        │ ← 속성
│번호 │      │      │              │
├─────┼──────┼──────┼──────────────┤
│C001 │홍길동│VIP   │010-1234-5678 │ ← 투플
│C002 │김철수│Gold  │010-2345-6789 │
│C003 │이영희│Silver│010-3456-7890 │
│C004 │박민수│VIP   │NULL          │
└─────┴──────┴──────┴──────────────┘

 차수(Degree): 4
 카디널리티(Cardinality): 4
```

## 2. 기본 용어

| 용어 | 설명 | 파일 시스템 대응 |
|:---|:---|:---|
| 릴레이션 (Relation) | 2차원 테이블 구조 | 파일 |
| 속성 (Attribute) | 릴레이션의 열 | 필드 |
| 투플 (Tuple) | 릴레이션의 행 | 레코드 |
| 도메인 (Domain) | 속성이 가질 수 있는 값의 집합 | 데이터 타입 |
| 널 (NULL) | 값을 모르거나 해당 없음 | - |
| 차수 (Degree) | 속성의 개수 (정적) | - |
| 카디널리티 (Cardinality) | 투플의 개수 (동적) | - |

### 도메인 예시

```text
고객번호: { 'C001', 'C002', ... }   -- 'C' + 3자리
이름:    { 문자열, 최대 50자 }
등급:    { VIP, Gold, Silver, Bronze }
연락처:  { 전화번호 형식 또는 NULL }
```

{{< callout type="info" >}}
**원자 값(Atomic Value)**: 관계 데이터 모델에서 속성 값은 더 이상 분해할 수 없는 원자 값만 사용할 수 있다. 이를 만족하면 제1정규형(1NF)이다.
{{< /callout >}}

## 3. 릴레이션 스키마와 인스턴스

```text
릴레이션 스키마 (Schema) = 논리적 구조
────────────────────────────────────
 고객(고객번호, 이름, 등급, 연락처)
 특징: 정적 (자주 안 변함)
 별칭: 릴레이션 내포 (Intension)

릴레이션 인스턴스 (Instance) = 실제 데이터
────────────────────────────────────
 (C001, 홍길동, VIP, 010-1234-5678)
 (C002, 김철수, Gold, 010-2345-6789)
 ...
 특징: 동적 (삽입·삭제·수정으로 변함)
 별칭: 릴레이션 외연 (Extension)
```

### 데이터베이스 구성

```text
데이터베이스 스키마
├── 고객(고객번호, 이름, 등급, 연락처)
├── 상품(상품번호, 상품명, 가격, 재고)
├── 주문(주문번호, 고객번호, 주문일, 총액)
└── 주문상세(주문번호, 상품번호, 수량)

데이터베이스 인스턴스
├── 고객 릴레이션 인스턴스 (4 투플)
├── 상품 릴레이션 인스턴스 (100 투플)
├── 주문 릴레이션 인스턴스 (500 투플)
└── 주문상세 릴레이션 인스턴스 (1200 투플)
```

## 4. 릴레이션의 4가지 특성

| 특성 | 설명 | 의미 |
|:---|:---|:---|
| 투플의 유일성 | 중복 투플 불가 | 키로 구별 가능 |
| 투플의 무순서 | 행 순서 무의미 | 내용으로 검색 |
| 속성의 무순서 | 열 순서 무의미 | 이름으로 접근 |
| 속성의 원자성 | 단일 값만 허용 | 1NF 만족 |

### 투플의 유일성

```text
올바름                          잘못됨
┌─────┬──────┬─────┐            ┌─────┬──────┬─────┐
│C001 │홍길동│VIP  │            │C001 │홍길동│VIP  │
│C002 │김철수│Gold │            │C002 │김철수│Gold │
│C003 │이영희│Silver│           │C001 │홍길동│VIP  │ ← 중복!
└─────┴──────┴─────┘            └─────┴──────┴─────┘
```

### 속성의 원자성

```text
잘못됨 (다중 값)
┌─────┬──────┬─────────────────┐
│C001 │홍길동│010-1234,02-1234 │ ← 원자성 위반
└─────┴──────┴─────────────────┘

올바름 (정규화)
┌─────┬──────┬─────────┬──────────┐
│C001 │홍길동│휴대폰   │010-1234  │
│C001 │홍길동│집       │02-1234   │
└─────┴──────┴─────────┴──────────┘
```

## 5. 키 (Key)

키는 릴레이션에서 투플을 유일하게 구별하는 속성 또는 속성들의 집합이다.

### 키의 특성

| 특성 | 설명 |
|:---|:---|
| 유일성 (Uniqueness) | 키 값은 투플마다 달라야 함 |
| 최소성 (Minimality) | 꼭 필요한 최소 속성들로만 구성 |

### 5.1 키의 종류

학생 릴레이션을 예로 든다.

```text
학생
┌─────────┬──────────┬──────┬────────┬────┐
│학번     │주민번호  │이름  │학과    │학년│
├─────────┼──────────┼──────┼────────┼────┤
│2023001  │990101-1  │홍길동│컴퓨터  │ 1  │
│2023002  │000215-2  │김영희│경영    │ 1  │
│2022015  │010320-1  │이철수│컴퓨터  │ 2  │
└─────────┴──────────┴──────┴────────┴────┘
```

**슈퍼키**: 유일성만 만족하는 키
```text
{학번}, {주민번호}, {학번, 이름}, {학번, 주민번호, 이름}
```

**후보키**: 유일성 + 최소성 모두 만족
```text
{학번}     ← OK
{주민번호} ← OK
{학번, 이름} ← X (최소성 위반)
```

**기본키**: 후보키 중 기본으로 선택한 키. NULL 불가, 변경 최소, 단순한 속성 선호.
```text
학번 vs 주민번호 → 학번 선택 (짧고 안정)
```

**대체키**: 기본키로 선택되지 못한 후보키
```text
기본키: {학번}, 대체키: {주민번호}
```

**외래키**: 다른 릴레이션의 기본키를 참조하는 속성
```text
학생                         학과
┌──────┬──────┬──────┐       ┌──────┬────────┐
│학번  │이름  │학과코드│      │학과코드│학과명 │
├──────┼──────┼──────┤       ├──────┼────────┤
│23001 │홍길동│CS    │─────→│CS    │컴퓨터  │
│23002 │김영희│BA    │─────→│BA    │경영    │
└──────┴──────┴──────┘       └──────┴────────┘
       외래키                         기본키
```

### 5.2 키 종류 비교

| 종류 | 유일성 | 최소성 | 선택 | 특징 |
|:---|:---|:---|:---|:---|
| 슈퍼키 (Super Key) | O | X | - | 가장 넓은 개념 |
| 후보키 (Candidate Key) | O | O | - | 기본키 후보 |
| 기본키 (Primary Key) | O | O | O | NULL 불가, 대표 식별자 |
| 대체키 (Alternate Key) | O | O | X | 후보키지만 미선택 |
| 외래키 (Foreign Key) | △ | - | - | 다른 릴레이션 PK 참조 |

### 키 포함 관계

```text
┌────────────────────────┐
│        슈퍼키           │
│  ┌──────────────────┐  │
│  │     후보키        │  │
│  │ ┌──────┬───────┐ │  │
│  │ │기본키│대체키 │ │  │
│  │ └──────┴───────┘ │  │
│  └──────────────────┘  │
└────────────────────────┘
 외래키: 별도 개념 (다른 릴레이션 PK 참조)
```

### 5.3 자기 참조 외래키

외래키가 같은 릴레이션의 기본키를 참조할 수도 있다.

```text
사원 (자기 참조)
┌──────┬──────┬────────┐
│사원  │이름  │상사번호│
├──────┼──────┼────────┤
│E001  │김사장│NULL    │ ← 최상위
│E002  │이부장│E001    │───┐
│E003  │박과장│E002    │───┤ FK
│E004  │최대리│E003    │───┘
└──────┴──────┴────────┘
  ↑              │
  └──────────────┘
      자기 참조
```

## 6. 무결성 제약조건

무결성(Integrity)은 데이터가 정확하고 유효하게 유지된 상태를 말한다.

### 무결성 vs 보안

| 구분 | 대상 | 목적 |
|:---|:---|:---|
| 무결성 | 권한 있는 사용자 | 잘못된 요구로 인한 데이터 부정확 방지 |
| 보안 | 권한 없는 사용자 | 무단 접근으로부터 데이터 보호 |

### 제약조건 종류

| 제약조건 | 대상 | 규칙 | 사례 |
|:---|:---|:---|:---|
| 개체 무결성 (Entity) | 기본키 | NULL 불가, 유일 | 고객번호는 NULL일 수 없다 |
| 참조 무결성 (Referential) | 외래키 | 참조 값 존재 또는 NULL | 주문의 고객번호는 고객에 존재해야 한다 |
| 도메인 무결성 (Domain) | 속성 값 | 도메인 범위 내 | 학년은 1~4 사이 |

### 6.1 개체 무결성

기본키를 구성하는 모든 속성은 NULL 값을 가질 수 없다.

```text
올바름
┌─────┬──────┬─────┐
│C001 │홍길동│VIP  │ ← PK NOT NULL
│C002 │김철수│NULL │ ← 비키 속성은 NULL OK
└─────┴──────┴─────┘

위반
┌─────┬──────┬─────┐
│NULL │홍길동│VIP  │ ← PK NULL 위반!
└─────┴──────┴─────┘
```

복합 기본키도 마찬가지다.

```text
주문상세(주문번호, 상품번호, 수량)
복합 PK: (주문번호, 상품번호)

OK   : (O001, P001, 3)
위반 : (O001, NULL, 3)  ← 일부 NULL도 위반
위반 : (NULL, P001, 3)
```

### 6.2 참조 무결성

외래키 값은 반드시 참조되는 릴레이션의 기본키 값으로 존재하거나 NULL이어야 한다.

```text
학과 (참조됨)                 학생 (참조함)
┌──────┬────────┐             ┌──────┬──────┬──────┐
│학과  │학과명  │             │학번  │이름  │학과  │
├──────┼────────┤             ├──────┼──────┼──────┤
│CS    │컴퓨터  │ ←─── OK ────│23001 │홍길동│CS    │
│BA    │경영    │ ←─── OK ────│23002 │김영희│BA    │
│EE    │전자    │              │23003 │이철수│NULL  │ OK
└──────┴────────┘              │23004 │박민수│CH    │ 위반!
                               └──────┴──────┴──────┘
```

### 6.3 도메인 무결성

속성 값은 해당 도메인에 정의된 범위 안에 있어야 한다.

```sql
학년 INT CHECK (학년 BETWEEN 1 AND 4)
성별 CHAR(1) CHECK (성별 IN ('M','F'))
등급 VARCHAR(10) CHECK (등급 IN ('VIP','Gold','Silver'))
```

### 참조 무결성 위반 시 대응

| 옵션 | 동작 |
|:---|:---|
| RESTRICT | 거부 (기본값) |
| CASCADE | 참조 투플도 함께 삭제/수정 |
| SET NULL | 외래키를 NULL로 변경 |
| SET DEFAULT | 외래키를 기본값으로 변경 |

```sql
CREATE TABLE 학생 (
    학번 VARCHAR(10) PRIMARY KEY,
    이름 VARCHAR(50) NOT NULL,
    학과코드 VARCHAR(10),
    FOREIGN KEY (학과코드) REFERENCES 학과(학과코드)
        ON DELETE SET NULL
        ON UPDATE CASCADE
);
```

## 7. SQL로 보는 관계 데이터 모델

### 릴레이션 생성

```sql
-- 학과 (참조됨)
CREATE TABLE 학과 (
    학과코드 VARCHAR(10) PRIMARY KEY,
    학과명 VARCHAR(50) NOT NULL,
    위치 VARCHAR(100)
);

-- 학생 (참조함)
CREATE TABLE 학생 (
    학번 VARCHAR(10) PRIMARY KEY,           -- 기본키 (개체 무결성)
    주민번호 VARCHAR(14) UNIQUE,             -- 대체키
    이름 VARCHAR(50) NOT NULL,
    학과코드 VARCHAR(10),
    학년 INT CHECK (학년 BETWEEN 1 AND 4),  -- 도메인 무결성
    FOREIGN KEY (학과코드)                   -- 참조 무결성
        REFERENCES 학과(학과코드)
        ON DELETE SET NULL
        ON UPDATE CASCADE
);
```

### 무결성 위반 시나리오

```sql
-- 개체 무결성 위반
INSERT INTO 학생 (학번, 이름) VALUES (NULL, '홍길동');
-- Error: Column '학번' cannot be null

-- 참조 무결성 위반
INSERT INTO 학생 (학번, 이름, 학과코드)
VALUES ('2023001', '홍길동', 'INVALID');
-- Error: foreign key constraint fails

-- 도메인 무결성 위반
INSERT INTO 학생 VALUES ('2023001', '...', '홍길동', 'CS', 5);
-- Error: CHECK constraint '학년' failed

-- 정상 삽입
INSERT INTO 학과 VALUES ('CS', '컴퓨터공학', '공학관 301');
INSERT INTO 학생 VALUES ('2023001', '990101-1234567', '홍길동', 'CS', 1);
```

{{< callout type="warning" >}}
외래키 삭제 규칙을 선택할 때 CASCADE는 편리하지만 파급 범위가 크다. 결제·주문처럼 이력을 남겨야 하는 경우 SET NULL이나 RESTRICT를 먼저 고려하자.
{{< /callout >}}

## 핵심 정리

| 개념 | 설명 |
|:---|:---|
| 릴레이션 | 2차원 테이블 구조 |
| 투플 / 속성 | 행 / 열 |
| 차수 / 카디널리티 | 속성 수 / 투플 수 |
| 도메인 | 속성이 가질 수 있는 값의 집합 |
| 슈퍼키 / 후보키 | 유일성 / 유일성 + 최소성 |
| 기본키 / 대체키 | 선택된 후보키 / 미선택 후보키 |
| 외래키 | 다른 릴레이션 PK 참조 |
| 개체 무결성 | 기본키 NULL 불가 |
| 참조 무결성 | 외래키는 존재 값 또는 NULL |
| 도메인 무결성 | 속성 값은 도메인 범위 내 |
| 릴레이션 4특성 | 투플 유일/무순서, 속성 무순서/원자성 |

{{< callout type="info" >}}
**용어 정리**
- **릴레이션 (Relation)**: 2차원 테이블 구조의 데이터 집합
- **투플 (Tuple)**: 릴레이션의 행
- **속성 (Attribute)**: 릴레이션의 열
- **도메인 (Domain)**: 속성이 가질 수 있는 값의 집합
- **차수 (Degree)**: 속성의 개수
- **카디널리티 (Cardinality)**: 투플의 개수
- **슈퍼키 (Super Key)**: 유일성을 만족하는 속성 집합
- **후보키 (Candidate Key)**: 유일성 + 최소성 만족
- **기본키 (Primary Key)**: 대표로 선택된 후보키, NULL 불가
- **대체키 (Alternate Key)**: 기본키로 선택되지 못한 후보키
- **외래키 (Foreign Key)**: 다른 릴레이션의 기본키를 참조
- **개체 무결성**: 기본키는 NULL일 수 없음
- **참조 무결성**: 외래키는 참조 값이 존재하거나 NULL
- **도메인 무결성**: 속성 값은 정의된 도메인 내에 있어야 함
- **원자 값 (Atomic Value)**: 더 이상 분해할 수 없는 단일 값
- **릴레이션 내포 / 외연**: 스키마 / 인스턴스의 별칭
