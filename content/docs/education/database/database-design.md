## 1. 데이터베이스 설계의 개념

데이터베이스 설계는 사용자의 요구사항을 분석하여, 그 요구를 정확하고 효율적으로 지원할 수 있는 데이터베이스를 만들어 내는 과정이다. 일반적으로 다음 5단계로 진행된다.

```text
[1] 요구사항 분석
      │  요구사항 명세서
      ▼
[2] 개념적 설계
      │  E-R 다이어그램
      ▼
[3] 논리적 설계
      │  릴레이션 스키마
      ▼
[4] 물리적 설계
      │  저장 구조·인덱스
      ▼
[5] 구현
      │  실제 데이터베이스
      ▼
```

### 단계별 개요

| 단계 | 주요 작업 | 결과물 | DBMS 의존성 |
|:---|:---|:---|:---|
| 1. 요구사항 분석 | 사용자 요구 수집·분석 | 요구사항 명세서 | 독립적 |
| 2. 개념적 설계 | E-R 다이어그램 작성 | 개념적 스키마 | 독립적 |
| 3. 논리적 설계 | 릴레이션 스키마 변환 | 논리적 스키마 | 모델 의존 |
| 4. 물리적 설계 | 저장 구조·인덱스 설계 | 물리적 스키마 | DBMS 의존 |
| 5. 구현 | DDL로 DB 생성 | 실제 DB | DBMS 의존 |

## 2. 1단계 요구사항 분석

데이터베이스에 대한 사용자들의 요구사항을 수집·분석하여 개발할 데이터베이스의 용도를 명확히 파악하는 단계다.

주요 작업은 다음과 같다.

- **사용자 범위 결정**: 주요 사용자와 역할을 정의한다
- **업무 분석**: 필요한 데이터와 처리 방식을 파악한다
- **요구 수집**: 인터뷰·설문·문서 분석으로 자료를 모은다
- **명세서 작성**: 분석 결과를 명확하게 문서화한다

### 요구사항 명세서 템플릿

체계적 분석을 위해 기능/비기능 요구사항을 분리해 기술한다.

```text
┌──────────────────────────────┐
│ 프로젝트: 온라인 서점         │
├──────────────────────────────┤
│ [1] 데이터 요구사항            │
│  - 고객, 도서, 주문, 주문상세  │
│                              │
│ [2] 기능 요구사항 (무엇을)      │
│  - 고객 가입/로그인           │
│  - 도서 검색/주문             │
│  - 주문 이력 조회             │
│                              │
│ [3] 비기능 요구사항 (어떻게)    │
│  - 성능: 응답시간 1초 이내     │
│  - 가용성: 99.9%             │
│  - 보안: 비밀번호 해시 저장    │
│  - 확장성: 연 30% 성장 대응    │
│                              │
│ [4] 제약 조건                 │
│  - 재고 0 이하 주문 불가       │
│  - 1주문 최대 1,000만 원      │
└──────────────────────────────┘
```

### 기능 vs 비기능 요구사항

| 구분 | 질문 | 예시 |
|:---|:---|:---|
| 기능 요구사항 | 시스템이 무엇을 하는가? | 회원 가입, 주문 처리, 검색 |
| 비기능 요구사항 | 얼마나 잘 하는가? | 성능, 보안, 가용성, 확장성 |

## 3. 2단계 개념적 설계

요구사항을 DBMS 독립적인 개념적 데이터 모델, 주로 E-R 모델로 표현하는 단계다.

### 개념적 모델링 절차

```text
요구사항 명세서
      │
      ▼
[1] 개체 추출 (명사)
      │
      ▼
[2] 속성 정의
      │
      ▼
[3] 키 속성 선별
      │
      ▼
[4] 관계 결정 (동사)
      │
      ▼
[5] E-R 다이어그램
```

### 개체 추출 예시

요구사항 명세서의 명사를 개체 후보로 선정한다.

```text
"고객은 여러 도서를 한 번에 주문할 수 있다"
  ↑            ↑              ↑
 개체         개체            개체
(고객)       (도서)          (주문)
```

### E-R 다이어그램 예시

```text
 ┌──────┐   주문    ┌──────┐   포함   ┌──────┐
 │ 고객 │──1:N──▶│ 주문 │──N:M──▶│ 도서 │
 └──┬───┘           └──┬───┘           └──┬───┘
    │                  │                  │
 고객번호(PK)      주문번호(PK)       도서번호(PK)
 이름              주문일             제목
 연락처            배송상태           저자
 주소              총금액             가격
                                     재고수량
```

## 4. 3단계 논리적 설계

개념적 스키마를 DBMS가 처리 가능한 논리적 스키마(릴레이션)로 변환하는 단계다.

### 변환 규칙 1: 개체 → 릴레이션

```text
개체 "고객"  →  릴레이션 "고객"

고객(고객번호, 이름, 연락처, 주소)
     ─────── PK
```

### 변환 규칙 2: 1:N 관계 → 외래키

N쪽 릴레이션에 1쪽의 기본키를 외래키로 추가한다.

```text
고객(1) ──< 주문 >── 주문(N)

주문(주문번호, 주문일, 총금액, 고객번호)
                              ────── FK
```

### 변환 규칙 3: N:M 관계 → 새 릴레이션

별도 릴레이션을 만들고 양쪽 기본키를 외래키로 가진다.

```text
주문(N) ──< 포함 >── 도서(M)

주문상세(주문번호, 도서번호, 수량)
          ─────── ─────── (복합 PK)
          FK       FK
```

### 변환 규칙 4: 1:1 관계 → 외래키

어느 한쪽에 상대 기본키를 외래키로 추가한다.

```text
직원(1) ──< 관리 >── 부서(1)

부서(부서번호, 부서명, 관리자사번)
                      ──────── FK
```

### 변환 결과: 릴레이션 스키마

```sql
고객(고객번호, 이름, 연락처, 주소, 가입일)
    PK: 고객번호

도서(도서번호, 제목, 저자, 출판사, 가격, 재고수량)
    PK: 도서번호

주문(주문번호, 주문일, 배송상태, 총금액, 고객번호)
    PK: 주문번호
    FK: 고객번호 → 고객(고객번호)

주문상세(주문번호, 도서번호, 수량, 단가)
    PK: (주문번호, 도서번호)
    FK: 주문번호 → 주문(주문번호)
    FK: 도서번호 → 도서(도서번호)
```

## 5. 4단계 물리적 설계

논리적 스키마를 기반으로, 실제 저장 장치에 효율적으로 저장하기 위한 내부 구조를 설계하는 단계다.

### 물리적 설계 요소

| 요소 | 설명 | 예시 |
|:---|:---|:---|
| 저장 구조 | 데이터 저장 방식 | 힙 파일, 순차 파일, 해시 파일 |
| 인덱스 | 빠른 검색을 위한 구조 | B-Tree, Hash, Bitmap |
| 파티셔닝 | 대용량 테이블 분할 | 범위, 해시, 리스트 |
| 클러스터링 | 관련 데이터를 물리적으로 근접 배치 | 클러스터 인덱스 |

### 인덱스 설계 기준

| 유리한 경우 | 불리한 경우 |
|:---|:---|
| WHERE에 자주 쓰이는 컬럼 | 자주 변경되는 컬럼 |
| JOIN에 쓰이는 외래키 | 선택도가 낮은 컬럼 (예: 성별) |
| ORDER BY / GROUP BY 컬럼 | 테이블 크기가 작은 경우 |
| 선택도(Selectivity)가 높은 컬럼 | 쓰기가 매우 잦은 컬럼 |

### 물리적 설계 예시

```sql
-- 인덱스 설계
CREATE INDEX idx_주문_고객번호 ON 주문(고객번호);
CREATE INDEX idx_주문_주문일   ON 주문(주문일);
CREATE INDEX idx_도서_제목     ON 도서(제목);

-- 연도별 파티셔닝
CREATE TABLE 주문 (
    주문번호 INT,
    주문일   DATE
) PARTITION BY RANGE (YEAR(주문일)) (
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p2025 VALUES LESS THAN (2026)
);
```

## 6. 5단계 구현

설계 결과를 DDL로 표현하여 실제 데이터베이스 객체를 생성하는 단계다.

```sql
-- 데이터베이스 생성
CREATE DATABASE 온라인서점;
USE 온라인서점;

-- 고객
CREATE TABLE 고객 (
    고객번호 VARCHAR(10)  PRIMARY KEY,
    이름     VARCHAR(50)  NOT NULL,
    연락처   VARCHAR(20),
    주소     VARCHAR(200),
    가입일   DATE         DEFAULT CURRENT_DATE
);

-- 도서
CREATE TABLE 도서 (
    도서번호 VARCHAR(13)  PRIMARY KEY,
    제목     VARCHAR(200) NOT NULL,
    저자     VARCHAR(100),
    출판사   VARCHAR(100),
    가격     INT          CHECK (가격 > 0),
    재고수량 INT          DEFAULT 0 CHECK (재고수량 >= 0)
);

-- 주문
CREATE TABLE 주문 (
    주문번호 VARCHAR(20)  PRIMARY KEY,
    주문일   DATETIME     DEFAULT CURRENT_TIMESTAMP,
    배송상태 VARCHAR(20)  DEFAULT '주문접수',
    총금액   INT          CHECK (총금액 <= 10000000),
    고객번호 VARCHAR(10)  NOT NULL,
    FOREIGN KEY (고객번호) REFERENCES 고객(고객번호)
        ON DELETE RESTRICT
        ON UPDATE CASCADE
);

-- 주문상세
CREATE TABLE 주문상세 (
    주문번호 VARCHAR(20),
    도서번호 VARCHAR(13),
    수량     INT NOT NULL CHECK (수량 > 0),
    단가     INT NOT NULL,
    PRIMARY KEY (주문번호, 도서번호),
    FOREIGN KEY (주문번호) REFERENCES 주문(주문번호)
        ON DELETE CASCADE,
    FOREIGN KEY (도서번호) REFERENCES 도서(도서번호)
        ON DELETE RESTRICT
);

-- 인덱스
CREATE INDEX idx_고객_이름      ON 고객(이름);
CREATE INDEX idx_주문_고객번호   ON 주문(고객번호);
CREATE INDEX idx_주문_주문일     ON 주문(주문일);
CREATE INDEX idx_도서_제목       ON 도서(제목);
```

## 7. 단계별 산출물 체크리스트

각 단계에서 다음 질문에 "예"라고 답할 수 있어야 다음 단계로 넘어갈 준비가 된 것이다.

### 요구사항 분석

- [ ] 주요 사용자와 업무 범위를 식별했는가?
- [ ] 데이터 요구사항과 처리 요구사항을 모두 문서화했는가?
- [ ] 성능·보안·가용성 등 비기능 요구사항을 포함했는가?
- [ ] 이해관계자 검토·승인을 거쳤는가?

### 개념적 설계

- [ ] 모든 개체가 식별되었는가?
- [ ] 각 개체에 적절한 속성이 배정되었는가?
- [ ] 모든 개체에 키 속성이 지정되었는가?
- [ ] 개체 간 관계와 카디널리티(1:1, 1:N, N:M)가 명확한가?

### 논리적 설계

- [ ] 모든 개체가 릴레이션으로 변환되었는가?
- [ ] N:M 관계가 별도 릴레이션으로 분해되었는가?
- [ ] 기본키와 외래키가 모두 정의되었는가?
- [ ] 정규화를 통해 이상(Anomaly)을 제거했는가?

### 물리적 설계

- [ ] 주요 조회 쿼리에 맞는 인덱스가 있는가?
- [ ] 대용량 테이블에 파티셔닝을 고려했는가?
- [ ] 백업·복구 전략을 수립했는가?
- [ ] 예상 데이터 증가량을 반영한 용량 산정을 했는가?

### 구현

- [ ] DDL이 논리·물리적 설계를 정확히 반영하는가?
- [ ] 제약조건이 누락 없이 선언되었는가?
- [ ] 테스트 데이터로 기능 요구사항을 검증했는가?
- [ ] 성능 요구사항(응답시간, 처리량)을 충족하는가?

## 8. 설계 품질 기준

| 기준 | 설명 | 확인 사항 |
|:---|:---|:---|
| 완전성 | 모든 요구사항 반영 | 필요한 데이터가 모두 포함되었는가? |
| 무결성 | 데이터 정확성 유지 | 제약조건이 적절히 정의되었는가? |
| 일관성 | 데이터 모순 없음 | 정규화가 적용되었는가? |
| 확장성 | 변경 용이성 | 구조 변경이 쉬운가? |
| 성능 | 효율적 접근 | 인덱스가 적절한가? |

## 9. 설계 방법 비교

| 방법 | E-R 모델 + 변환 규칙 | 정규화 |
|:---|:---|:---|
| 접근 방식 | Top-Down (하향식) | Bottom-Up (상향식) |
| 시작점 | 요구사항 명세서 | 기존 데이터·속성 |
| 주요 작업 | 개체·관계 도출 → 변환 | 함수 종속성 분석 → 분해 |
| 장점 | 전체 구조 파악 용이 | 이상 제거 보장 |
| 단점 | 대규모에서 복잡 | 전체 구조 파악 어려움 |

실무에서는 두 방법을 함께 사용한다. E-R 모델로 전체 구조를 먼저 잡고, 정규화로 이상 현상을 제거하는 식이다.

## 핵심 정리

| 개념 | 설명 |
|:---|:---|
| 설계 5단계 | 요구사항 → 개념 → 논리 → 물리 → 구현 |
| 요구사항 명세서 | 데이터·기능·비기능·제약 정리 |
| 개념적 설계 | E-R 다이어그램, DBMS 독립적 |
| 논리적 설계 | 릴레이션 변환 규칙 적용 |
| 변환 규칙 | 개체→릴레이션, 1:N→FK, N:M→새 릴레이션 |
| 물리적 설계 | 저장 구조·인덱스·파티셔닝 |
| 정규화 | 이상 현상 제거, 무결성 확보 |
| 좋은 설계 | 완전성·무결성·일관성·확장성·성능 |

{{< callout type="info" >}}
**용어 정리**
- **요구사항 명세서**: 시스템이 수행해야 할 기능·비기능·제약을 정리한 문서
- **기능 요구사항**: 시스템이 "무엇을" 하는지
- **비기능 요구사항**: 성능·보안·가용성 등 품질 속성
- **개념적 스키마**: DBMS 독립적, E-R 다이어그램 기반 모델
- **논리적 스키마**: 관계 모델 등 특정 모델에 맞춘 릴레이션 스키마
- **물리적 스키마**: 저장 구조·인덱스 등 DBMS에 특화된 설계
- **E-R 다이어그램**: 개체·속성·관계를 시각화한 표기법
